# New Era 开发步骤文档

> 基于 `PRD-NewEra.md` 与 `evolution/00-13` 系列文档整理。  
> 前端 UI 视觉稿由 Google Stitch 设计，本开发文档聚焦工程实现、接口联调、数据层、核心引擎与 Demo 验收。

---

## 1. 开发目标

New Era 当前阶段是竞赛 Demo，目标是在 3-5 分钟演示中完整展示：

1. JD URL 抓取与 JD 文本评估。
2. AI 岗位 6 维评估，支持 SSE 流式逐维输出。
3. 假 AI 岗风险识别，能区分真 AI 岗与培训贷/伪 AI 岗。
4. 简历解析、AI 技术栈画像、JD 匹配与优化建议。
5. 行业洞察与首页仪表盘，形成完整产品体验。

本阶段不做真实多用户登录、投递追踪、Offer 对比、自动同步招聘平台、RSS 自动抓取、复杂权限系统；登录/注册页仅作为前端演示入口，不接后端鉴权。

## 2. 技术基线

| 模块 | 技术选型 |
|------|----------|
| 应用框架 | Next.js 16 + React 19 + App Router |
| 样式 | Tailwind CSS 4 + CSS 变量 |
| 图标 | Lucide React |
| 动画 | Framer Motion |
| 数据库 | SQLite + better-sqlite3 |
| 输入校验 | zod |
| JD 抓取 | fetch + Cheerio |
| AI 模型 | DeepSeek V4 Pro，中转站 API |
| 流式输出 | SSE，前端用 `fetch()` + `ReadableStream` 消费 |

环境变量建议：

```env
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=
DEEPSEEK_MODEL=
```

## 4. 推荐开发顺序

### Phase 1：项目初始化

目标：搭好可运行的 Next.js 工程骨架。

任务：

1. 使用 Next.js 16 初始化项目。
2. 安装依赖：`better-sqlite3`、`zod`、`cheerio`、`lucide-react`、`framer-motion`。
3. 建立基础目录：

```text
src/
  app/
  components/
    ui/
    features/
  lib/
data/
demo-data/
```

4. 配置 `src/app/layout.tsx`、`src/app/globals.css`。
5. 接入 Google Stitch 输出的基础设计 token。

验收标准：

- `npm run dev` 可正常启动。
- 首页路由 `/` 可打开。
- 全局字体、背景色、基础布局生效。

---

### Phase 2：数据层与基础类型

目标：先完成稳定的数据读写能力，避免后续页面各自写数据逻辑。

任务：

1. 新建 `src/lib/server-db.ts`。
2. 使用 `better-sqlite3` 连接 `data/new-era.db`。
3. 启用：
   - `journal_mode = WAL`
   - `busy_timeout = 5000`
   - `foreign_keys = ON`
4. 自动建表：

```sql
reports(id, company, role, jd_text, score_json, risks_json, report_json, created_at)
resumes(id, name, content_json, skills_json, ecosystem_json, created_at, updated_at)
settings(id, key, value_json, updated_at)
```

5. 新建 `src/lib/schemas.ts`，用 zod 定义 JD、报告、简历、风险扫描输入输出。
6. 新建 `src/lib/types.ts`，沉淀前后端共享类型。

验收标准：

- 启动时自动创建数据库和表。
- 可通过临时脚本或 API 写入、读取 `reports`。
- 所有写入接口都经过 zod 校验。

---

### Phase 3：Demo 数据准备

目标：保障竞赛演示稳定，不依赖真实招聘网站或外部数据。

任务：

1. 新建 `demo-data/preset-jds.json`。
2. 准备 5-8 条预置 JD：
   - 真 AI 岗 ×3
   - 假 AI 岗 ×2
   - 边界/中风险岗位 ×1-2
3. 新建 `demo-data/insights.json`，保存行业趋势、热门技术栈、薪资基准、推荐公司。
4. 准备 1-2 份示例简历文本。

验收标准：

- 不调用外部 API 时，也能完整演示首页、评估页、风险页、行业洞察页。
- 每条 Demo JD 都有明确演示目的：高分、低分、风险拦截、边界判断。

---

### Phase 4：核心引擎开发

目标：优先完成产品最有差异化的能力。

#### 4.1 风险识别引擎

文件建议：

```text
src/lib/risk-engine.ts
src/lib/risk-rules.ts
```

任务：

1. 实现 20+ 条风险规则。
2. 支持严重度：`critical`、`high`、`medium`、`low`。
3. 输出风险信号：

```ts
{
  signal: string;
  excerpt: string;
  severity: "critical" | "high" | "medium" | "low";
  source: "regex" | "dictionary" | "llm";
}
```

4. 实现风险计分：
   - 命中 `critical`：虚假 AI 岗，综合评分强制不高于 1.5。
   - 总分 >= 6：高风险，评分上限 2.5。
   - 总分 2-5：中风险。
   - 总分 <= 1：低风险。
5. LLM 语义补充作为增强项，前两层规则必须可独立运行。

#### 4.2 岗位评估引擎

文件建议：

```text
src/lib/evaluate-engine.ts
src/lib/deepseek-client.ts
```

任务：

1. 解析 JD 中的公司、岗位、薪资、技术栈、岗位方向。
2. 生成 A-F 六维评估：
   - A 岗位定位，15%
   - B 技术栈匹配，25%
   - C 经验要求，20%
   - D 成长价值，15%
   - E 薪酬竞争力，10%
   - F 风险合规，15%
3. 输出结构化 JSON，方便报告页渲染。
4. 结合风险引擎结果修正综合评分。
5. 保存完整报告到 SQLite `reports` 表。

#### 4.3 简历解析与匹配引擎

文件建议：

```text
src/lib/resume-parser.ts
src/lib/resume-match.ts
```

任务：

1. 解析简历文本为结构化 JSON。
2. 提取技能、项目、年限、当前方向。
3. 计算 AI 生态位向量：
   - LLM 应用层
   - AI Infra
   - 算法研究
   - 具身智能
   - AI 安全
4. 计算简历与 JD 的方向匹配度、技能覆盖率、综合匹配度。
5. 输出技能缺口与简历优化建议。

验收标准：

- 风险引擎本地规则不依赖 LLM 也能给出结果。
- 评估引擎能稳定输出 6 个维度。
- 简历匹配能输出匹配度、缺口、优化建议。

### Phase 5：API 路由开发

目标：为页面提供稳定接口。

优先级 P0：

| API | 方法 | 说明 |
|-----|------|------|
| `/api/health` | GET | 健康检查 |
| `/api/jd/fetch` | POST | JD URL 抓取 |
| `/api/scan-risks` | POST | 风险扫描 |
| `/api/evaluate` | POST | 同步评估 fallback |
| `/api/evaluate/stream` | POST | SSE 流式评估 |
| `/api/reports` | GET | 评估历史列表 |
| `/api/reports/[id]` | GET | 单份报告 |

优先级 P1：

| API | 方法 | 说明 |
|-----|------|------|
| `/api/resume/parse` | POST | 简历解析 |
| `/api/resume/match` | POST | 简历匹配 |
| `/api/resume/optimize` | POST | 简历优化建议 |
| `/api/insights/trends` | GET | 行业趋势数据 |
| `/api/insights/brief` | POST | AI 市场周报 |
| `/api/settings` | GET/PUT | 用户设置读写 |

不列入本阶段后端范围：

| API | 原因 |
|-----|------|
| `/api/auth/*` | 登录/注册页仅为前端演示态，竞赛 Demo 不做真实多用户鉴权 |

SSE 事件格式：

```json
{ "type": "phase", "phase": "scanning_risks" }
{ "type": "risks", "risks": [] }
{ "type": "phase", "phase": "evaluating", "section": "A" }
{ "type": "section", "section": "A", "data": {} }
{ "type": "score", "overall": 4.2 }
{ "type": "done", "report_id": 1 }
```

验收标准：

- 所有 API 有统一错误格式。
- `/api/evaluate/stream` 能逐段推送。
- SSE 失败时前端可降级调用 `/api/evaluate`。
- API Key 只在服务端读取，不出现在前端 bundle。

### Phase 6：页面与组件落地

目标：把 Google Stitch 设计稿转成可联调的 Next.js 页面。

基础组件：

```text
components/ui/
  Button.tsx
  Card.tsx
  Badge.tsx
  Progress.tsx
  Textarea.tsx
  StatCard.tsx
  EmptyState.tsx
  ErrorState.tsx
```

业务组件：

```text
components/features/
  Sidebar.tsx
  EvaluateStream.tsx
  RiskPanel.tsx
  ReportSection.tsx
  ResumeMatchPanel.tsx
  InsightCharts.tsx
```

页面顺序：

1. `/evaluate`：JD 评估入口，优先完成。
2. `/evaluate/report`：评估报告页（当前 Demo 路由；后续可扩展为动态详情页）。
3. `/risks`：假 AI 岗风险扫描页。
4. `/`：首页仪表盘。
5. `/resume`：简历管理与匹配页。
6. `/insights`：行业洞察页。

支撑页面：

7. `/reports`：评估历史页。
8. `/settings`：用户设置页。
9. `/auth`：登录/注册前端演示页。

核心功能页面必须实现：

- 初始态。
- 加载态。
- 成功态。
- 错误态。
- 空数据态。
- Demo 数据快速入口。

支撑页面至少需要初始态、交互反馈、禁用/空状态，以及清晰的“当前为前端演示或后续接入”边界说明。

验收标准：

- 6 个核心页面与支撑页面视觉统一。
- 主要按钮、表单、卡片、状态提示与 Stitch 设计一致。
- 页面之间路由可闭环：评估页 -> 报告页 -> 首页/历史。

### Phase 7：联调与异常处理

目标：确保演示时不因为外部 API 或抓取失败中断。

任务：

1. JD URL 抓取失败时允许手动粘贴。
2. DeepSeek 调用失败时展示错误提示和重试按钮。
3. SSE 断连自动重试，最多 2 次。
4. SSE 重试失败后调用同步评估接口。
5. SQLite 写入失败时不导致页面白屏。
6. Demo 数据入口永远可用。

验收标准：

- 断网或 DeepSeek 不可用时，页面仍能展示预置 Demo 流程。
- 所有错误都有用户可理解的提示。
- 控制台无明显运行时错误。

## 6. 最小可演示闭环

最小 Demo 闭环建议按以下路径实现：

```text
首页
  -> 点击“评估一个 AI 岗位”
  -> 进入 JD 评估页
  -> 选择预置真 AI 岗
  -> SSE 流式输出 A-F 六维评估
  -> 自动跳转报告页
  -> 返回风险扫描页
  -> 选择预置假 AI 岗
  -> 展示风险信号与低评分
```

这个闭环覆盖竞赛 Demo 的核心差异化：AI 专属评估 + 流式体验 + 假 AI 岗拦截。

## 7. 开发注意事项

- 不要把 DeepSeek API Key 写到前端代码或提交到仓库。
- API 返回结构必须稳定，前端不要依赖 LLM 原始文本解析页面。
- 预置 Demo 数据必须可脱离外部 API 运行。
- 风险识别优先用规则和字典，LLM 只做补充。
- 数据库只通过 `server-db.ts` 写入，避免多处直接操作 SQLite。
- Stitch 输出只作为 UI 参考，最终组件仍应拆成可维护的 React 组件。
- 评估报告保存结构化 JSON，Markdown 文本只能作为展示补充。
- PDF 导出优先使用浏览器打印，Playwright PDF 可作为后续增强。
