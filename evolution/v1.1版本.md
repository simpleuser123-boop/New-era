# New Era v1.1 开始阶段开发顺序

> 目标：把竞赛 demo 中已有但未完整落地的入口，先补成可用的产品闭环。v1.1 开始阶段只覆盖 1-8，不涉及招聘平台授权、真实投递、公网部署、支付订阅。

## 范围边界

- 不做招聘网站自动爬取。
- 不做真实岗位平台 API 授权。
- 不做一键投递到招聘平台。
- 不做公网分享链接。
- 不做支付、订阅、多用户 SaaS。
- 数据优先存储在本地 SQLite。

## 开发顺序总览

| 顺序 | 功能 | 类型 | 目标产出 |
|---:|---|---|---|
| 1 | 报告页导出 PDF | 单页补洞 | `window.print()` + 打印样式 |
| 2 | 报告页分享改为复制报告摘要 | 单页补洞 | 一键复制报告摘要到剪贴板 |
| 3 | 新增最小 `applications` 表 | 数据基础 | 本地投递记录表 |
| 4 | 报告页加入投递清单 | 动作闭环 | 从报告生成一条 `待投递` 记录 |
| 5 | 简历表增强字段 | 数据基础 | 支持方向、默认、归档 |
| 6 | 简历管理默认简历 | 单页补洞 | 一键设为默认简历 |
| 7 | 简历管理删除 / 归档 | 单页补洞 | 删除确认或软归档 |
| 8 | 简历管理方向标签 | 单页补洞 | 按 AI 方向标记简历 |

---

## 1. 报告页导出 PDF

### 目标

让报告页顶部的“导出 PDF”按钮成为真实可用功能。v1.1 开始阶段优先使用浏览器打印，不做服务端 PDF。

### 实现范围

- 给“导出 PDF”按钮绑定 `window.print()`。
- 增加 print CSS：
  - 隐藏侧边栏、顶部导航、按钮区。
  - 保留报告主体、评分、风险摘要、关键词、优化建议。
  - 控制页面背景、边距、分页。
- 避免打印时内容被截断。

### 不做

- 不做 `/api/reports/[id]/pdf`。
- 不做 Playwright 服务端生成 PDF。

### 验收标准

- 点击“导出 PDF”能打开浏览器打印面板。
- 打印预览只包含报告主体。
- 页面按钮、侧边栏和无关导航不出现在打印内容里。

### 生成系统提示词要点

- 目标文件：报告页组件和全局 CSS。
- 要求：只实现浏览器打印导出，不新增服务端 PDF API。
- 验收：点击按钮触发打印，打印样式干净。

## 2. 报告页分享改为复制报告摘要

### 目标

由于 v1.1 不做公网部署，报告页“分享报告”不做外链分享，改成“复制摘要”。

### 实现范围

- 将按钮文案从“分享报告”调整为“复制摘要”或“复制报告摘要”。
- 点击后复制结构化文本到剪贴板：
  - 公司
  - 岗位
  - 综合分
  - 风险等级
  - 推荐结论
  - 关键技术栈
  - 主要建议
- 成功后显示短暂成功反馈。
- 复制失败时显示友好错误。

### 不做

- 不做公网报告链接。
- 不做社交平台分享。
- 不做二维码。

### 验收标准

- 点击按钮后剪贴板中有可读的中文报告摘要。
- 成功和失败都有明确反馈。
- 未加载报告时按钮不可误操作。

### 生成系统提示词要点

- 目标文件：报告页组件。
- 要求：用 Clipboard API 复制摘要，不做公网分享。
- 验收：复制内容完整、格式适合发给自己或朋友。

## 3. 新增最小 `applications` 表

### 目标

建立本地投递记录表，为“加入投递清单”和后续 Tracker / Analytics 做数据基础。

### 建议表结构

```sql
CREATE TABLE IF NOT EXISTS applications (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  report_id INTEGER,
  resume_id INTEGER,
  company TEXT NOT NULL,
  role TEXT NOT NULL,
  source_url TEXT,
  status TEXT NOT NULL DEFAULT 'to_apply',
  score_overall REAL,
  risk_level TEXT,
  recommended_resume_name TEXT,
  notes TEXT NOT NULL DEFAULT '',
  tags_json TEXT NOT NULL DEFAULT '[]',
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY(report_id) REFERENCES reports(id),
  FOREIGN KEY(resume_id) REFERENCES resumes(id)
);
```

### 状态枚举

v1.1 开始阶段只保留最小状态：

- `to_apply`：待投递
- `applied`：已投递
- `abandoned`：已放弃

### 实现范围

- 在 SQLite 自动迁移中创建 `applications` 表。
- 增加基础类型和 schema 校验。
- 增加最小 DB 方法：
  - `createApplication`
  - `listApplications`
  - `getApplicationById`
  - `updateApplicationStatus`

### 不做

- 不做复杂状态机。
- 不做批量操作。
- 不做招聘平台状态同步。

### 验收标准

- 应用启动后自动创建表。
- 能写入一条 `to_apply` 记录。
- 能读取列表。
- 能更新状态。

### 生成系统提示词要点

- 目标文件：`server-db.ts`、`schemas.ts`、`types.ts`，必要时新增 API route。
- 要求：只做最小 `applications` 数据能力。
- 验收：本地 SQLite 能新增、读取、改状态。

## 4. 报告页加入投递清单

### 目标

把报告页原来的“立即投递简历”改成“加入投递清单”，避免无授权的一键投递，同时形成本地产品闭环。

### 实现范围

- 按钮文案改为“加入投递清单”。
- 点击后调用本地 API 创建 `applications` 记录。
- 默认状态为 `to_apply`。
- 写入字段：
  - `report_id`
  - `company`
  - `role`
  - `score_overall`
  - `risk_level`
  - `recommended_resume_name`（如可取得）
  - `notes` 初始为空
- 成功后提示“已加入投递清单”。
- 避免重复加入同一报告，可提示“已在投递清单中”。

### 不做

- 不跳转招聘平台。
- 不自动投递。
- 不同步外部投递状态。

### 验收标准

- 从报告页能创建一条本地投递记录。
- 重复点击不会产生无意义重复记录。
- 成功后用户明确知道下一步是手动去平台投递。

### 生成系统提示词要点

- 目标文件：报告页组件、`applications` API。
- 要求：按钮只写入本地投递清单，不做真实投递。
- 验收：报告能转成 `to_apply` 投递记录。

## 5. 简历表增强字段

### 目标

补齐多简历管理需要的基础字段，为默认简历、方向标签、归档做准备。

### 建议新增字段

```sql
ALTER TABLE resumes ADD COLUMN direction TEXT NOT NULL DEFAULT 'general';
ALTER TABLE resumes ADD COLUMN is_default INTEGER NOT NULL DEFAULT 0;
ALTER TABLE resumes ADD COLUMN archived_at TEXT;
```

### 方向枚举

- `general`：通用
- `llm_application`：LLM 应用层
- `ai_infra`：AI Infra
- `algorithm_research`：算法研究
- `embodied_ai`：具身智能
- `ai_safety`：AI 安全

### 实现范围

- 自动迁移兼容已有数据库。
- 更新 Resume 类型。
- 更新保存、读取、列表映射逻辑。
- 确保旧数据默认 `direction = general`。

### 不做

- 不做简历版本管理。
- 不做复制简历。
- 不做文件上传增强。

### 验收标准

- 老数据不丢失。
- 新保存简历包含方向和默认状态。
- 归档字段存在但不影响旧流程。

### 生成系统提示词要点

- 目标文件：`server-db.ts`、`schemas.ts`、`types.ts`。
- 要求：兼容旧 SQLite 数据，不能破坏已有简历读取。
- 验收：旧简历可读，新简历有 direction / is_default / archived_at。

## 6. 简历管理默认简历

### 目标

支持用户设置一份默认简历，评估和匹配流程优先使用它。

### 实现范围

- 简历列表中展示默认标识。
- 增加“设为默认”操作。
- 设置某份简历为默认时，其他简历自动取消默认。
- 评估页或匹配页读取默认简历时，优先使用 `is_default = 1`。

### 不做

- 不做多用户默认简历。
- 不做按岗位自动切换默认简历。

### 验收标准

- 任意时刻最多只有一份默认简历。
- 默认简历在页面上有明确标识。
- 重新设置默认简历后 UI 和数据库一致。

### 生成系统提示词要点

- 目标文件：简历 DB 方法、API、简历页面组件。
- 要求：默认简历唯一。
- 验收：设置 A 为默认后，B 自动取消默认。

## 7. 简历管理删除 / 归档

### 目标

让用户能清理不再使用的简历。v1.1 开始阶段优先软归档，避免误删数据。

### 实现范围

- 简历列表增加“归档”或“删除”入口。
- 点击后显示确认。
- 软归档：写入 `archived_at`。
- 默认列表隐藏已归档简历。
- 如果归档的是默认简历：
  - 自动取消默认。
  - 可选择最近一份未归档简历设为默认，或提示用户重新设置。

### 不做

- 不做回收站完整页面。
- 不做批量删除。
- 不做物理删除，除非后续明确需要。

### 验收标准

- 用户确认后简历从默认列表消失。
- 数据库仍保留记录。
- 默认简历被归档时不会留下无效默认状态。

### 生成系统提示词要点

- 目标文件：简历 API、DB 方法、简历列表组件。
- 要求：优先软归档，不物理删除。
- 验收：归档后列表隐藏，数据库保留，默认状态正确。

## 8. 简历管理方向标签

### 目标

让用户给不同简历标记 AI 方向，支持后续更精准的 JD 匹配和投递建议。

### 实现范围

- 简历创建 / 保存时可选择方向。
- 简历列表展示方向标签。
- 简历详情或解析结果中可以修改方向。
- 匹配页展示“简历方向 vs 岗位方向”的提示。

### 不做

- 不做复杂标签系统。
- 不做多方向权重编辑。
- 不做自动生成多份方向简历。

### 验收标准

- 每份简历都有一个方向。
- 方向标签在列表和匹配流程中可见。
- 修改方向后能保存并刷新展示。

### 生成系统提示词要点

- 目标文件：简历页面组件、保存 API、schema。
- 要求：用固定方向枚举，不做自由文本标签。
- 验收：方向可选择、可保存、可展示。
