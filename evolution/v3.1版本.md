# New Era v3.1 修复与收口开发顺序

> 目标：在不突破 local-first 和安全边界的前提下，收口 v3 已完成能力，修复当前回归、兼容层、验收文档和关键体验问题，把 New Era 从“功能已具备”推进到“可稳定演示、可持续迭代、可明确验收”的状态。v3.1 不以新增大功能为目标，重点做修复、统一、收口和证据补齐。

## 范围边界

- 继续保持 local-first：业务数据优先存储在本地 `data/new-era.db`。
- 不做招聘网站自动爬取。
- 不做真实招聘平台 API 授权。
- 不做自动投递，不模拟已向外部平台提交简历。
- 不做自动联系 HR、自动发邮件、自动站内信。
- 不做支付、订阅、多用户 SaaS。
- 不做公开分享链接。
- 不做无人监管的自治任务队列。
- 不做无限长期记忆。
- Query 工具保持只读；Action 工具仍然只能创建待确认 proposal，确认前不得写业务表。
- 不把 demo 数据伪装成真实用户数据，不把模型增强伪装成确定结论。

## 当前基础

v3 结束后，当前仓库已经具备以下能力：

- v2 主功能闭环：JD 评估、风险扫描、报告持久化、Resume、Tracker、Career DNA、Analytics、Interview、Settings、数据导出。
- v2.1 本地 owner 鉴权：注册、登录、登出、受保护 API、受保护页面、账号安全设置。
- v3 Agent 主链路：工具注册表、Query Tools、子 Agent 路由、Action proposal、`/api/agent/run` SSE、流式 `/assistant` UI、会话与可控记忆。
- 自动化脚本：`npm run lint`、`npm run build`、`npm run smoke`、`npm run smoke:auth`、`npm run smoke:agent` 和各子 smoke 脚本。
- 浏览器验收资产：`output/v3-agent-acceptance/assistant-desktop.png`、`output/v3-agent-acceptance/assistant-narrow.png` 已存在。

## 当前暴露的问题

以下问题已在当前仓库中可复现或可直接观察到：

1. 默认 `npm run smoke` 可能在旧 `/api/assistant` 的本地回答检查中超时。
   - `src/app/api/assistant/route.ts` 中 `ASSISTANT_MODEL_TIMEOUT_MS = 22_000`
   - `scripts/smoke-support.mjs` 中默认 `SMOKE_REQUEST_TIMEOUT_MS = 10_000`
   - 结果是默认 smoke 可能误报失败，延长到 `30000` 后可通过。
2. 旧 `/api/assistant` 与新 `/api/agent/run` 双栈并存，职责边界还不够清晰。
3. `docs/v3-agent-acceptance-checklist.md` 仍有多处“待填写”，但对应截图与 smoke 证据已存在。
4. `docs/v2-acceptance-checklist.md` 仍未回填，和当前代码状态脱节。
5. `src/components/features/history/AssessmentHistoryWorkspace.tsx` 删除报告仍使用 `window.confirm`，不符合当前可控确认交互的方向。
6. demo fallback、preset data、真实本地数据的边界虽然多数文案已控制，但仍缺少统一收口和显式分层。
7. `tmp/collect-risk-benchmark-candidates.mjs` 仍有 lint warning，影响全量质量信号。

## 37. Smoke 与兼容层超时修复

### 目标

让默认 `npm run smoke` 在当前仓库和默认配置下稳定通过，不依赖手工加大超时环境变量；同时明确旧 `/api/assistant` 在模型慢、不可用或无 key 时的 fallback 行为。

### 必须做

1. 复盘 `scripts/smoke.mjs` 中 `/api/assistant` 检查的真实需求，是验证“能回答”，还是验证“本地 fallback 可用”。
2. 修正默认 smoke 超时策略，使 `npm run smoke` 在默认配置下稳定通过。
3. 视代码结构选择其一：
   - 调整旧 `/api/assistant` 的模型超时与 fallback 策略；
   - 或让 smoke 显式命中本地 fallback；
   - 或把旧 Assistant 检查改为可控的 Agent 路径。
4. 增加说明文档，解释为什么默认 smoke 现在能稳定通过。

### 不做

- 不重写 v3 Agent Loop。
- 不新增新 Agent。
- 不做 Step 38-44。

### 验收标准

- `npm run smoke` 默认直接通过。
- 不需要人工额外设置 `SMOKE_REQUEST_TIMEOUT_MS=30000` 才能通过。
- `/api/assistant` 在模型不可用或超时场景下有明确、可测试的 fallback。

## 38. Assistant 双栈收口

### 目标

明确旧 `/api/assistant` 与新 `/api/agent/run` 的职责关系，避免后续继续双倍维护输入校验、fallback、上下文拼装和 smoke。

### 建议方向

优先考虑以下三种方案之一：

1. 兼容层方案：
   - 保留 `/api/assistant` 作为旧接口；
   - 内部复用 v3 Agent 的只读能力或统一上下文；
   - 前端逐步切到 `/api/agent/run`。
2. 收缩方案：
   - `/assistant` 页面全面使用 v3 Agent；
   - `/api/assistant` 仅保留最小兼容逻辑，文档标记为 legacy。
3. 显式分层方案：
   - `/api/assistant` 只做一次性本地摘要；
   - `/api/agent/run` 承担流式、多 Agent、工具调用和会话记忆。

### 必须做

1. 写清接口职责、调用边界和后续迁移策略。
2. 避免同一页面同时维护两套回答范式。
3. 统一 smoke 的主验证路径。
4. 更新相关文档和注释，避免后续误判旧接口仍是主链路。

### 不做

- 不扩展到招聘平台授权。
- 不做新的模型供应商接入。
- 不做 Step 39-44。

### 验收标准

- `/assistant` 的主链路清晰。
- v2/v3 文档不再互相冲突。
- smoke 和浏览器验收不会在旧/新两套链路之间来回漂移。

## 40. Demo / Real 数据边界统一

### 目标

让用户始终知道当前页面和结果来自：

- 真实本地 SQLite 数据
- 预置 demo 数据
- 本地规则 fallback
- 联网搜索结果
- 模型增强结果

避免“结果能看，但不知道是不是自己的真实数据”。

### 必须做

1. 盘点下列区域的数据来源提示是否一致：
   - `/evaluate`
   - `/evaluate/report`
   - `/reports`
   - `/insights`
   - `/resume`
   - `/assistant`
   - `/analytics`
   - `/settings` 导出说明
2. 对 fallback、demo、preset、mock、联网结果建立统一的文案和视觉标识规则。
3. 避免把 `preset_demo_data`、`demo_fallback` 等内部实现名直接裸露给用户。
4. 导出与说明文案中清楚区分“本地真实数据”和“预置演示资产”。

### 不做

- 不新增云同步。
- 不做真实市场级数据平台。
- 不做 Step 41-44。

### 验收标准

- 用户在任何关键页面都能判断当前数据来源。
- Demo 与真实本地数据不会混淆。
- Analytics 不会把个人样本误包装成行业统计。

## 41. 高风险交互确认统一

### 目标

把删除报告、清除数据、忽略 proposal 等高风险或不可逆操作统一到项目内可见、可控、可恢复的确认交互，不再依赖原生 `window.confirm`。

### 当前已知问题

- `src/components/features/history/AssessmentHistoryWorkspace.tsx` 删除报告仍使用 `window.confirm`。

### 必须做

1. 盘点当前项目内所有高风险确认入口。
2. 统一确认交互模式：
   - 明确对象
   - 明确影响范围
   - 明确是否可恢复
   - 明确确认/取消状态反馈
3. 先替换报告删除的原生确认。
4. 确保桌面和窄屏都可操作。

### 不做

- 不引入复杂全局弹窗系统，除非当前代码结构确实需要。
- 不顺带重构无关页面。
- 不做 Step 42-44。

### 验收标准

- 项目关键写操作不再依赖原生 `alert/confirm`。
- 删除类操作有一致的确认体验。
- 窄屏不遮挡、不错位、不溢出。

## 42. 真假 AI 岗识别质量收口

### 目标

继续强化 New Era 的核心差异化：真/假 AI 岗识别，不靠泛化产品功能堆砌，而是靠更稳定、更可解释、更可校准的风险判断能力。

### 建议方向

1. 基准集治理：
   - 维护 `demo-data/risk-benchmark-jds.jsonl`
   - 标注真 AI、假 AI、边界岗
2. 反馈闭环：
   - 把用户在报告页提交的风险反馈沉淀成可回看记录
   - 区分误报、漏报、严重度不准
3. 规则版本与复盘：
   - 每次重要规则调整都能复盘 benchmark 变化
   - 记录 recall / precision / risk level accuracy 的变化
4. 解释增强：
   - 让“为什么判成假 AI 岗”比“打了几分”更清晰

### 必须做

1. 梳理当前 benchmark、反馈、报告页解释是否已形成闭环。
2. 补齐缺失的质量度量和回看路径。
3. 不让规则迭代直接静默改变历史信号的可解释性。

### 不做

- 不接真实招聘平台数据管道。
- 不做自动训练或黑盒自学习。
- 不做 Step 43-44。

### 验收标准

- 真假 AI 岗识别有可复跑 benchmark。
- 误报/漏报反馈能进入规则迭代输入。
- 规则迭代后能说明性能变化，而不是只凭主观感受。

   - 或迁移到更合适的脚本目录。
3. 不让临时采集脚本影响主工程质量判断。

### 不做

### 验收标准

- 临时脚本与正式脚本边界更清楚。
