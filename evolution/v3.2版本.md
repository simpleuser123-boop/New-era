# New Era v3.2 岗位验真体验增强开发顺序

> 目标：在不新增三个左侧导航入口、不改变 local-first 和安全边界的前提下，把 New Era 已有的“真假 AI 岗识别”能力升级成评委和真实用户都能快速理解、验证并采取行动的岗位验真体验。

## 阶段定位

v3.2 不新增泛化求职功能，也不重写现有风险引擎。它只围绕三个高价值能力展开：

1. Benchmark 可视化质量看板：证明识别能力可复跑、可度量，不是只依赖提示词。
2. 真假 AI 岗对决：让用户在同一视图中看到真 AI、假 AI 和边界岗的证据差异。
3. 风险驱动的 HR 核实问题：把风险判断转化为用户可以继续验证岗位的行动。

三个能力共同形成：

```text
看见差异 -> 相信判断 -> 采取行动
```

## 信息架构决定

### 左侧导航

不新增三个导航项，也不增加独立的“Benchmark”“真假对决”“反向提问”入口。

当前左侧已有：

- `风险扫描` -> `/risks`
- `面试教练` -> `/interview`
- `评估历史` -> `/reports`

v3.2 继续复用现有 `风险扫描` 入口：

```text
/risks
  -> 单岗验真
  -> 真假对决
  -> 识别质量
```

- “单岗验真”和“真假对决”使用页面内分段控制器或 Tab。
- “识别质量”作为 `/risks` 页面中的只读质量面板，不进入主导航。
- “生成核实问题”作为风险结果和评估报告中的上下文动作，不进入主导航。
- 左侧 `AppSidebar` 的 `navItems` 数量保持不变。

### 页面关系

```text
/risks
  ├─ 单岗验真：保留当前扫描流程
  ├─ 真假对决：并排比较两个预置 JD 的识别结果
  └─ 识别质量：展示本地 benchmark 指标、样本构成和规则版本

/evaluate/report
  └─ 风险结果后提供“生成核实问题”

/interview
  └─ 继续负责求职者回答训练，不与“向 HR 核实岗位”混为同一功能
```

## 范围边界

- 继续保持 local-first。
- 不接招聘平台 API，不抓取招聘网站。
- 不自动向 HR 发送消息、邮件或站内信。
- 不自动投递。
- 不新增公开分享链接。
- 不把本地 benchmark 指标包装成行业统计。
- 不把 Demo JD 包装成真实实时招聘数据。
- 不做自动训练或黑盒自学习。
- 用户反馈不能自动修改风险规则、评分或历史报告。
- Query 保持只读；Action 仍然只能创建待确认 proposal。
- 不重写 `src/lib/risk-engine.ts` 的核心判断逻辑。
- 不为竞赛展示复制一套与正式产品分叉的风险算法。

## 当前基础

当前仓库已经具备：

- `/risks` 单岗位风险扫描和真/假示例。
- `POST /api/scan-risks`。
- `src/lib/risk-engine.ts` 的本地确定性风险扫描。
- `src/lib/semantic-risk-engine.ts` 的语义补充能力。
- `demo-data/preset-jds.json` 中的真 AI、假 AI、边界岗预置数据。
- `demo-data/risk-benchmark-jds.jsonl` 中的 100 条本地 benchmark 样本。
- `npm run smoke:risk-benchmark` 和机器可读结果。
- 风险规则版本记录。
- `/api/risk-feedback` 和误报、漏报、严重度不准反馈。
- `/evaluate/report` 中的风险解释、原文证据和反馈入口。
- `/interview` 中的本地问题模板、模型增强和 fallback。

v3.2 应复用以上基础，不重复建立第二套数据、规则或面试系统。

## v3.2 开发顺序总览

| 顺序 | 功能 | 类型 | 目标产出 |
|---:|---|---|---|
| 45 | Benchmark 只读服务与质量看板 | 技术可信度 | `/risks` 可展示样本构成、核心指标、规则版本和限制说明 |
| 46 | 真假 AI 岗对决体验 | 核心差异化 | `/risks` 增加真假对决模式，并排展示结论和原文证据 |
| 47 | 风险驱动的 HR 核实问题 | 产品行动闭环 | 风险结果和报告可生成可复制的岗位核实问题 |
| 48 | v3.2 集成、竞赛演示与回归验收 | 发布质量 | 三项能力串成稳定演示路径，并完成桌面、窄屏和安全边界验收 |

## 依赖关系

```text
45 Benchmark 只读服务与质量看板
  -> 46 真假 AI 岗对决体验
      -> 47 风险驱动的 HR 核实问题
          -> 48 v3.2 集成、竞赛演示与回归验收
```

- Step 45 先建立可信指标读取和展示契约，避免 Step 46 在页面里硬编码指标。
- Step 46 再完成评委最容易理解的核心对比体验。
- Step 47 基于已稳定的风险结果生成下一步核实动作。
- Step 48 最后串联演示路径并完成回归，不在前三步顺带做发布收口。

## 45. Benchmark 只读服务与质量看板

### 目标

把当前只能通过命令和文档查看的风险 benchmark，转成产品内可读取、可解释、不可误导的质量看板。

### 产品位置

- 放在 `/risks` 页面内部。
- 建议使用“识别质量”折叠区、抽屉或页面区段。
- 不新增左侧导航项。
- 不单独创建面向用户的顶层 `/benchmark` 导航入口。

### 建议技术结构

建议新增或调整：

- `src/lib/risk-benchmark.ts`
- `src/lib/types.ts`
- `src/lib/schemas.ts`
- `src/app/api/risk-benchmark/route.ts`
- `src/components/features/risks/RiskBenchmarkPanel.tsx`
- `scripts/smoke-risk-benchmark.mjs`
- 可新增 `scripts/smoke-risk-benchmark-api.mjs`
- `package.json`
- `docs/v3-2-step45-risk-benchmark-dashboard.md`

### 数据契约

`GET /api/risk-benchmark` 返回只读结果，至少包含：

```ts
type RiskBenchmarkOverview = {
  dataset: {
    total: number;
    realAi: number;
    fakeAi: number;
    borderline: number;
    source: "local_benchmark";
  };
  ruleVersion: string;
  generatedAt: string;
  metrics: {
    riskLevelAccuracy: number;
    decisionAccuracy: number;
    fakeAiRecall: number;
    fakeAiPrecision: number;
    realAiFalsePositiveRate: number;
    borderlinePrecision: number;
    expectedRiskSignalHitRate: number;
  };
  issues: {
    correct: number;
    severityMismatch: number;
    borderlineFalsePositive: number;
    missingExpectedSignal: number;
  };
  disclaimer: string;
};
```

字段可根据现有 benchmark 结果调整，但必须使用 Zod 校验，并保持 API 返回稳定。

### 必须做

1. 从现有 `smoke-risk-benchmark.mjs` 中梳理可复用的核心计算逻辑。
2. 把 benchmark 计算或结果映射收口到服务端共享模块，避免 API 和 smoke 各自维护一套指标公式。
3. API 直接读取本地 benchmark 数据并生成只读结果。
4. API 不依赖 `tmp/risk-benchmark-result.json` 才能工作。
5. API 不写 SQLite，不修改 benchmark 数据集，不触发模型调用。
6. 页面展示：
   - 样本总量与真 AI、假 AI、边界岗构成；
   - decision accuracy；
   - fake AI recall；
   - fake AI precision；
   - real AI false positive rate；
   - 当前风险规则版本；
   - 已知误差分布。
7. 页面明确显示：
   - 这是本地标注 benchmark；
   - 不代表招聘市场总体统计；
   - 不代表对所有岗位达到相同准确率；
   - 指标会随规则版本变化。
8. 指标加载失败时，风险扫描主功能仍然可用。

### 交互要求

- 看板默认保持紧凑，不抢占单岗验真的首屏主要操作。
- 使用数字、短标签和简短限制说明，不使用营销式夸大文案。
- 指标使用百分比时明确保留位数。
- 桌面可使用紧凑网格，窄屏改为两列或单列。
- 不使用饼图堆叠过多标签。
- 不通过颜色单独表达好坏，必须同时提供文字。

### 不做

- 不接真实招聘平台数据。
- 不展示未经验证的行业渗透率、市场规模或用户成功率。
- 不做在线模型评测平台。
- 不做 benchmark 样本编辑后台。
- 不做自动规则优化。
- 不做 Step 46-48。

### 验收标准

- `GET /api/risk-benchmark` 返回经过 schema 校验的本地只读结果。
- API 读取失败时返回明确错误，不暴露本机绝对路径。
- `npm run smoke:risk-benchmark` 继续通过。
- 新 API smoke 覆盖指标字段、样本数量、规则版本和只读边界。
- `/risks` 能看到质量看板。
- 页面明确标记本地 benchmark 边界。
- 桌面和 `390x844` 窄屏无横向溢出。
- `AppSidebar` 未新增导航项。

## 46. 真假 AI 岗对决体验

### 目标

让用户在同一页面、同一规则版本下并排查看两个 JD 的真假判断、风险等级和原文证据，直观理解真 AI、假 AI 和边界岗的差异。

### 产品位置

继续使用 `/risks`。

建议在当前页面标题下增加分段控制器：

```text
[ 单岗验真 ] [ 真假对决 ]
```

- 默认仍为“单岗验真”，不破坏现有用户路径。
- “真假对决”是同一页面的另一种工作模式。
- Benchmark 看板在两种模式下都可访问，但不占主要操作位。

### 建议技术结构

建议新增或调整：

- `src/components/features/risks/RiskScanPage.tsx`
- `src/components/features/risks/RiskScanWorkspace.tsx`
- `src/components/features/risks/RiskComparisonWorkspace.tsx`
- `src/components/features/risks/RiskComparisonSelector.tsx`
- `src/components/features/risks/RiskComparisonResult.tsx`
- `src/components/features/risks/riskScanModel.ts`
- `demo-data/preset-jds.json`
- 可新增 `scripts/smoke-risk-comparison.mjs`
- `package.json`
- `docs/v3-2-step46-risk-comparison.md`

### 对决场景

第一版提供三个稳定预置组合：

1. 真 AI 岗 vs 假 AI 岗。
2. 真 AI 岗 vs 边界岗。
3. 假 AI 岗 vs 边界岗。

预置组合必须复用 `demo-data/preset-jds.json` 中已有且标明数据来源的记录。

### 必须做

1. 保留当前单岗扫描功能和默认行为。
2. 新增页面内模式切换，不修改左侧导航。
3. 对决模式允许选择两个不同的预置 JD。
4. 两侧必须调用同一套风险扫描契约：
   - 优先调用 `POST /api/scan-risks`；
   - 不复制前端风险判断逻辑；
   - 不为演示硬编码最终结论。
5. 可以并行请求两侧扫描，但必须分别处理成功、失败和 fallback。
6. 每侧至少展示：
   - 公司与岗位；
   - 真 AI 岗倾向、边界岗或假 AI 岗风险；
   - 风险等级；
   - 主要风险信号；
   - 命中原文证据；
   - 建议动作；
   - 数据来源；
   - 规则版本。
7. 中间或结果顶部展示“关键差异”，但差异必须来自两侧真实扫描结果。
8. 支持“一键交换左右岗位”。
9. 切回单岗验真时，不自动把对决结果写入报告、Tracker 或历史记录。
10. API 失败且使用 preset fallback 时，必须明确标记 fallback，不能伪装成实时扫描成功。

### 布局要求

- 桌面端使用稳定的双列布局，两侧宽度一致。
- 窄屏按“岗位 A -> 岗位 B -> 差异总结”纵向排列。
- 不把整个页面做成嵌套卡片。
- 原文证据需要可读，不得只展示风险分数。
- 两侧标题、结论和操作区尺寸稳定，避免异步结果导致布局大幅跳动。
- 使用现有 Lucide 图标和设计 token。

### 数据可信度

- 预置对决数据必须标记为竞赛 Demo 仿真数据或本地 benchmark 样本。
- 不声称当前正在从招聘网站实时抓取。
- 不声称真/假判断具有法律结论。
- 使用“风险倾向”“需要核实”“建议谨慎”等产品文案。

### 不做

- 不做任意招聘网站 URL 的双页面抓取。
- 不做外部平台岗位搜索。
- 不做公开分享对决链接。
- 不自动保存对决结果。
- 不在本步骤增加 HR 核实问题。
- 不做 Step 47-48。

### 验收标准

- `/risks` 默认单岗验真行为不回归。
- 页面内可切换到真假对决。
- 三组预置组合均可运行。
- 两侧使用相同扫描 API 和规则版本。
- 至少一个真/假组合能清晰展示原文证据差异。
- 一侧失败时另一侧结果仍可阅读。
- fallback 标记正确。
- 桌面和 `390x844` 窄屏无重叠、溢出或不可读文本。
- `AppSidebar` 仍只有现有 `风险扫描` 入口。

## 47. 风险驱动的 HR 核实问题

### 目标

把“这个岗位有风险”转化为“用户下一步应该向 HR 或面试官核实什么”，让风险判断进入可执行的求职决策。

### 产品位置

主要入口：

- `/risks` 单岗扫描结果。
- `/risks` 真假对决的每侧结果。
- `/evaluate/report` 的风险解释区域。

按钮建议使用：

```text
生成核实问题
```

不要使用：

- 自动联系 HR
- 一键发送
- 自动询问招聘方

### 与面试教练的边界

- “HR 核实问题”用于用户向招聘方验证岗位真实性和职责边界。
- `/interview` 现有问题用于训练求职者如何回答面试问题。
- 两者可以复用部分风险上下文，但不能混成同一组问题。
- 不修改现有 Interview Session 的问题含义。

### 建议技术结构

建议新增或调整：

- `src/lib/risk-verification-questions.ts`
- `src/lib/types.ts`
- `src/lib/schemas.ts`
- `src/app/api/risk-verification-questions/route.ts`
- `src/components/features/risks/RiskVerificationQuestions.tsx`
- `src/components/features/risks/RiskScanResults.tsx`
- `src/components/features/risks/RiskComparisonResult.tsx`
- `src/components/features/evaluate/report/EvaluationReportView.tsx`
- 可新增 `scripts/smoke-risk-verification-questions.mjs`
- `package.json`
- `docs/v3-2-step47-risk-verification-questions.md`

### 第一版生成策略

第一版优先采用本地确定性生成，不依赖模型才能工作。

问题模板根据风险类别生成，例如：

| 风险类别 | 核实方向 |
|---|---|
| 培训贷 / 收费 | 是否需要支付培训费、服务费、押金；是否签署培训或借款协议 |
| 招转培 | 当前流程是正式招聘还是培训筛选；培训结束是否保证劳动合同 |
| 传统岗位包装 AI | 日常工作中模型、RAG、Agent 或算法任务的实际占比 |
| 职责边界不清 | 所属团队、直接负责人、前三个月交付物和考核指标 |
| 过度承诺 | 薪资、offer、入职时间和承诺是否写入正式合同 |
| 技术含金量不足 | 是否有生产环境、真实用户、评测指标和模型迭代闭环 |
| 数据标注边界 | 岗位核心是标注执行、数据运营，还是模型和算法开发 |

### 返回契约

`POST /api/risk-verification-questions` 接收：

```ts
type RiskVerificationQuestionInput = {
  jdText: string;
  company?: string;
  role?: string;
};
```

服务端必须重新执行或复用可信风险扫描，不接受客户端直接提交“已判定风险”作为唯一依据。

返回：

```ts
type RiskVerificationQuestionResult = {
  decisionLabel: string;
  riskLevel: string;
  ruleVersion: string;
  questions: Array<{
    id: string;
    category: string;
    question: string;
    reason: string;
    relatedEvidence?: string;
    priority: "high" | "medium" | "low";
  }>;
  disclaimer: string;
};
```

字段可按现有类型调整，但必须经过 Zod 校验。

### 必须做

1. 建立纯函数生成器，根据风险信号和岗位证据输出问题。
2. 相同输入、相同规则版本应得到稳定结果。
3. 高危风险问题排在前面。
4. 每个问题说明“为什么要问”，并尽量关联原文证据。
5. 如果没有明显高风险，也应生成验证 AI 技术含金量和团队边界的问题。
6. 提供：
   - 单条复制；
   - 复制全部；
   - 复制成功和失败反馈。
7. 问题只显示在页面，不自动保存到长期数据。
8. 不创建 Agent Action Proposal，因为“复制问题”不是业务写入。
9. 不调用外部联系方式，不自动发送。
10. API 和 UI 明确说明问题用于辅助核实，不代表招聘方一定存在违规。

### 模型策略

第一版不要求 DeepSeek。

如果后续增加模型润色：

- 本地问题必须先生成。
- 模型只允许改写表达或补充追问。
- 模型不可删除高优先级安全问题。
- 模型不可声称已经联系 HR。
- 模型不可写数据库。
- 模型不可用时必须返回完整本地结果。

### 不做

- 不自动发送消息。
- 不读取通讯录。
- 不接邮箱、短信或招聘平台。
- 不保存 HR 联系方式。
- 不创建聊天机器人替用户谈判。
- 不做 Step 48。

### 验收标准

- 高风险、边界岗和低风险 JD 都能生成合理问题。
- 培训贷和招转培必须生成收费、协议、劳动关系核实问题。
- 传统岗位包装必须生成 AI 工作占比、交付物和技术闭环问题。
- 无模型、无 API key 时功能完整可用。
- API 输入过短、过长或非法时返回明确错误。
- API 不写 reports、applications、settings、sessions 等业务表。
- 单条复制和复制全部可用。
