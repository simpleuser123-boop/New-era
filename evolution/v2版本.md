# New Era v2 产品路线与分步开发提示词

> 目标：把 New Era 从“本地 JD 评估 + 简历匹配 + 投递清单”升级为更完整的个人 AI 求职 OS。v2 重点补齐 Tracker、Career DNA、Analytics、Assistant、Interview、Resume Judge、数据治理和验收体系。每个编号都应拆成一个独立新对话执行，不要在同一个对话里顺带实现后续步骤。

## 范围边界

- 不做招聘网站自动爬取。
- 不做真实招聘平台 API 授权。
- 不做自动投递，不模拟已向外部平台提交简历。
- 不做自动联系 HR、自动发邮件、自动站内信。
- 不做支付、订阅、多用户 SaaS。
- 不做公开分享链接。
- 不做复杂 BI 系统，只做个人求职复盘所需分析。
- 不做无限长期记忆，所有记忆和画像都必须可查看、可编辑、可删除。
- 数据继续优先存储在本地 SQLite。
- Agent 可以辅助决策，但所有写操作必须让用户确认。

## 当前基础

v2 默认已经具备 v1.1 和 v1.2 的以下能力：

- JD 评估、SSE 流式评估、风险扫描、报告持久化。
- 简历解析、简历保存、默认简历、方向标签、软归档。
- 简历和 JD 匹配，能输出匹配度、技能覆盖和缺口建议。
- 行业趋势页面和 DeepSeek 周报生成。
- 本地 `applications` 表。
- `/applications` 页面、列表、搜索、状态筛选。
- 投递状态：`to_apply / applied / interviewing / offer_received / rejected / abandoned`。
- 投递详情：备注、标签、原岗位链接、下次跟进时间。
- 首页投递漏斗统计。
- 本地 `settings` 表。

## 14. Tracker 完整工作台

### 目标

把 v1.2 的本地投递清单升级为完整求职跟进工作台。Tracker 负责记录用户手动维护的投递过程，不连接招聘平台，不自动替用户投递。

### 用户场景

- 用户从报告页加入投递清单后，能继续记录投递渠道、优先级和跟进动作。
- 用户每天打开 Tracker，能看到哪些岗位逾期未跟进。
- 用户能从列表或看板视图快速判断当前求职管道是否健康。
- 用户能打开某条投递记录，看到从加入清单到投递、面试、Offer 或放弃的时间线。

### 数据结构建议

在 `applications` 基础上补充字段：

```sql
ALTER TABLE applications ADD COLUMN priority TEXT NOT NULL DEFAULT 'medium';
ALTER TABLE applications ADD COLUMN channel TEXT;
ALTER TABLE applications ADD COLUMN applied_at TEXT;
ALTER TABLE applications ADD COLUMN interview_at TEXT;
ALTER TABLE applications ADD COLUMN offer_at TEXT;
ALTER TABLE applications ADD COLUMN rejected_at TEXT;
ALTER TABLE applications ADD COLUMN abandoned_at TEXT;
```

新增事件表：

```sql
CREATE TABLE IF NOT EXISTS application_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  application_id INTEGER NOT NULL,
  event_type TEXT NOT NULL,
  title TEXT NOT NULL,
  note TEXT NOT NULL DEFAULT '',
  happened_at TEXT NOT NULL DEFAULT (datetime('now')),
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY(application_id) REFERENCES applications(id)
);

CREATE INDEX IF NOT EXISTS idx_application_events_application_id
  ON application_events(application_id, happened_at DESC);
```

建议枚举：

```ts
type ApplicationPriority = "high" | "medium" | "low";

type ApplicationEventType =
  | "created"
  | "status_changed"
  | "note_added"
  | "follow_up_scheduled"
  | "interview_scheduled"
  | "offer_received"
  | "rejected"
  | "abandoned";
```

### API 建议

- `GET /api/applications`
  - 支持 `q`、`status`、`priority`、`channel`、`limit`。
- `GET /api/applications/[id]`
  - 返回投递详情和 `events`。
- `PATCH /api/applications/[id]`
  - 支持状态、优先级、渠道、备注、标签、链接、下次跟进时间。
  - 状态变化时自动写入 `application_events`。
- `GET /api/applications/[id]/events`
  - 返回时间线。
- `POST /api/applications/[id]/events`
  - 手动新增备注、跟进、面试安排等事件。
- `GET /api/applications/stats`
  - 增加逾期跟进数、优先级分布、渠道分布。

### 页面与交互

- `/applications` 保留列表视图，同时新增看板视图。
- 顶部增加：
  - 搜索
  - 状态筛选
  - 优先级筛选
  - 渠道筛选
  - 列表 / 看板切换
- 详情面板增加：
  - 优先级
  - 投递渠道
  - 关键日期
  - 时间线
  - 新增事件表单
- 逾期跟进需要明显标记，但不要制造夸张焦虑。

### 不做

- 不接招聘平台状态同步。
- 不自动判断用户是否真的投递。
- 不自动发送跟进消息。
- 不做拖拽看板库，开始阶段可以用分组列表模拟看板。
- 不做日历集成。

### 验收标准

- 用户能设置优先级和渠道，刷新后仍然保存。
- 状态变化会生成时间线事件。
- 用户能手动新增一条跟进事件。
- 下次跟进时间早于当前时间时，页面显示逾期状态。
- 看板视图能按状态展示投递记录。
- 首页或 stats API 能返回逾期跟进数量。
- 页面没有任何“系统已替你投递”的误导文案。

## 15. Career DNA 求职画像

### 目标

建立用户本地职业画像，作为 Tracker、Analytics、Assistant 和 Interview 的长期上下文。Career DNA 是结构化求职目标和偏好，不是性格测试，不是外部平台画像，也不是自动长期记忆代理。

### 用户场景

- 用户能明确自己的目标 AI 方向、岗位、城市、工作模式和薪资底线。
- 用户能记录自己的核心优势、短板、不可接受条件和当前求职策略。
- 系统可以基于默认简历、最近报告和 Tracker 生成画像建议，但必须由用户确认后保存。
- 后续 Analytics 和 Assistant 可以用 Career DNA 判断哪些岗位更符合目标。

### 数据结构建议

优先复用 `settings` 表，新增 key：`career_profile`。

```ts
type CareerProfileSettings = {
  targetDirections: ResumeDirection[];
  targetRoles: string[];
  targetCities: string[];
  preferredWorkModes: Array<"remote" | "hybrid" | "onsite" | "flexible">;
  salaryExpectation?: {
    minMonthlyK?: number;
    maxMonthlyK?: number;
    negotiable?: boolean;
  };
  strengths: Array<{
    title: string;
    evidence?: string;
  }>;
  weaknesses: Array<{
    title: string;
    mitigation?: string;
  }>;
  dealBreakers: string[];
  preferredCompanyStages: string[];
  currentStrategy?: string;
  generatedSuggestions?: Array<{
    id: string;
    source: "default_resume" | "reports" | "tracker";
    title: string;
    detail: string;
    accepted: boolean;
  }>;
  updatedAt: string;
};
```

### API 建议

- `GET /api/career-profile`
  - 返回已保存画像、默认值和可选建议。
- `PUT /api/career-profile`
  - 保存用户确认后的画像。
- `POST /api/career-profile/suggestions`
  - 从默认简历、最近报告、Tracker 生成建议。
  - 只返回建议，不自动覆盖。

也可以先复用 `/api/settings`，但建议新增独立 API，避免设置页越来越臃肿。

### 页面与交互

新增页面或设置页独立模块：

- 推荐路由：`/career`
- 侧边栏入口：“Career DNA”或“求职画像”
- 页面模块：
  - 目标方向
  - 目标岗位
  - 目标城市
  - 工作模式
  - 薪资期望
  - 核心优势
  - 能力短板
  - 不接受条件
  - 当前策略
  - 系统建议卡片

系统建议必须提供：

- 来源说明。
- 接受按钮。
- 忽略按钮。
- 不自动写入保存值。

### 不做

- 不做外部社交平台画像抓取。
- 不做性格画像。
- 不做用户未确认的长期记忆写入。
- 不把简历解析结果等同于 Career DNA。
- 不做多账号画像。

### 验收标准

- 用户可以打开 Career DNA 页面。
- 用户可以编辑并保存目标、优势、短板和偏好。
- 刷新后数据仍然存在。
- 默认简历存在时，可以生成画像建议。
- 系统建议不会自动覆盖用户已保存画像。
- 空数据状态可用。
- API 输入有 Zod 校验。

## 16. Analytics 求职分析

### 目标

基于本地报告、简历、Tracker 和 Career DNA，提供个人求职过程分析。Analytics 用于帮助用户复盘，不做招聘平台真实转化率同步，也不承诺全网市场统计。

### 用户场景

- 用户想知道当前求职漏斗是否健康。
- 用户想知道哪些 AI 方向评分高、风险低、匹配高。
- 用户想知道常见技能缺口是什么。
- 用户想知道哪些岗位逾期未跟进。
- 用户想知道当前推进岗位是否符合 Career DNA。

### 分析维度

| 模块 | 指标 |
|---|---|
| 漏斗分析 | 已评估、加入清单、待投递、已投递、面试中、Offer、未通过、已放弃 |
| 方向分析 | 不同 AI 方向评估数量、平均分、风险比例 |
| 风险分析 | 高风险岗位占比、常见风险信号 |
| 匹配分析 | 高匹配岗位占比、技能缺口 Top |
| Tracker 分析 | 不同状态数量、逾期跟进、优先级分布、渠道分布 |
| 简历效果 | 默认简历关联岗位数量、方向匹配情况 |
| Career DNA 匹配 | 当前推进岗位与目标方向、城市、工作模式的匹配度 |

### API 建议

开始阶段可先做一个综合接口：

```text
GET /api/analytics
```

返回结构建议：

```ts
type AnalyticsOverview = {
  generatedAt: string;
  sampleSize: {
    reports: number;
    resumes: number;
    applications: number;
  };
  funnel: Array<{ key: string; label: string; count: number }>;
  directions: Array<{ direction: string; count: number; averageScore: number; highRiskCount: number }>;
  riskSignals: Array<{ label: string; count: number; severity?: string }>;
  skillGaps: Array<{ skill: string; count: number; priority: "quick_fix" | "long_term" }>;
  tracker: {
    overdueFollowUps: number;
    byPriority: Array<{ priority: string; count: number }>;
    byChannel: Array<{ channel: string; count: number }>;
  };
  careerFit?: {
    summary: string;
    matchedCount: number;
    warningCount: number;
  };
  caveats: string[];
};
```

后续可以拆分：

- `GET /api/analytics/funnel`
- `GET /api/analytics/directions`
- `GET /api/analytics/risks`
- `GET /api/analytics/gaps`

- 新增 `/analytics` 页面。
- 侧边栏入口：“求职分析”。
- 页面至少包含：
  - 求职漏斗

  - AI 方向分布
  - 风险信号 Top

### 不做

- 不做招聘平台真实转化率同步。
- 不做复杂 BI 报表。
- 不做薪资市场报告。

### 验收标准

## 17. Assistant Agent MVP

### 目标

新增一个基于本地数据的求职 Assistant，让用户能用自然语言查询报告、Tracker、Career DNA 和 Analytics，并获得下一步行动建议。v2 只做单 Agent MVP，不做复杂多 Agent 编排。

### 用户场景

- “我现在最该跟进哪几个岗位？”
- “哪些岗位最符合我的 Career DNA？”
- “帮我解释这个报告为什么风险高。”
- “这周求职漏斗怎么样？”
- “帮我为这个岗位准备面试问题。”
- “把这个岗位改成已投递。”

### Agent 原则

- 回答必须引用本地数据来源。
- 不确定的信息要明确说不确定。
- 可以生成行动建议，但不能声称已经联系外部平台。
- 写操作必须二次确认。
- Assistant 是统一入口，但底层仍复用已有 API 和 DB 方法。

### 可用工具范围

- Tracker 成为完整求职跟进模块，包含优先级、渠道、关键日期、事件时间线、看板和逾期提醒。
- Career DNA 可编辑、可保存，并能从默认简历、报告和 Tracker 生成待确认建议。
- Analytics 能展示本地求职漏斗、方向、风险、技能缺口、逾期跟进和 Career DNA 匹配。
- Assistant 能基于本地数据回答问题，输出引用来源和待确认动作。
- Interview Coach 能基于报告和简历生成问题、评分回答，并保存 STAR 故事。
- Resume Judge 能生成多模式简历优化建议，支持 ATS 检查和采纳/拒绝反馈。
- 用户能从设置页导出本地数据，并理解本地数据和模型 API 的边界。
- 项目拥有 smoke 脚本和 v2 人工验收清单。
- `npm run lint` 和 `npm run build` 通过。
- 不出现自动投递、自动联系 HR、真实平台授权、全网趋势承诺等误导文案。

## 一个步骤一个新对话的执行方式

每个新对话只实现一个编号，不要在同一对话顺带实现后续步骤。

推荐依次创建 8 个新对话：

1. `New Era v2 - Step 14 - Tracker 完整工作台`
2. `New Era v2 - Step 15 - Career DNA 求职画像`
3. `New Era v2 - Step 16 - Analytics 求职分析`
4. `New Era v2 - Step 17 - Assistant Agent MVP`
5. `New Era v2 - Step 18 - Interview Coach MVP`
6. `New Era v2 - Step 19 - Resume Judge v2`
7. `New Era v2 - Step 20 - 数据导出与隐私治理`
8. `New Era v2 - Step 21 - 自动化验收与发布清单`

每个对话的基础要求：

- 先阅读本文件和前置阶段文档。
- 只实现当前编号，后续步骤不要提前做。
- 先检查现有代码，复用已有类型、API、组件和设计系统。
- 兼容已有本地 SQLite 数据，新增字段必须通过非破坏性自动迁移。
- 所有输入输出都要经过 Zod 或等价 schema 校验。
- 涉及页面时要覆盖加载、空状态、错误状态和窄屏布局。
- 完成后运行 `npm run lint` 和 `npm run build`。
- 涉及页面时做一次浏览器验收。
- 最终回复中列出修改文件、验收结果和下一步应该开启的对话标题。
