# New Era v2.1 登录鉴权开发顺序

> 目标：把当前 `/auth` 的前端 Demo 升级为可用的本地登录鉴权能力，用来保护本机 SQLite 中的个人求职数据。v2.1 默认做“本地单用户 / owner 账号保护”，不是云端多用户 SaaS，也不接招聘平台授权。

## 范围边界

- 做真实注册、登录、登出、会话校验和受保护页面入口。
- 做本地账号表、会话表、密码哈希、HttpOnly Cookie。
- 做 API 级鉴权，确保本地报告、简历、投递、画像、面试等数据接口不能被未登录请求直接访问。
- 做页面级访问控制：未登录访问产品页时回到 `/auth`。
- 继续保持 local-first：数据仍优先存储在本地 `data/new-era.db`。
- 第一版只支持一个本地 owner 账号。注册成功后，后续注册入口应提示已存在本地账号。
- 不做短信验证码服务。
- 不做微信、GitHub 或其他第三方 OAuth。
- 不做邮箱找回密码。
- 不做云端同步、远程账户中心、支付、订阅、多租户隔离。
- 不做招聘平台 API 授权，不自动投递，不自动联系 HR。
- 不自动清空或重写已有本地业务数据。

## 当前基础

- `/auth` 页面已存在，但只是前端模拟登录/注册。
- `AuthCard` 当前通过 `setTimeout` 模拟提交成功，没有调用后端鉴权接口。
- 当前依赖中没有 `next-auth`、`bcrypt`、JWT 等鉴权依赖。
- 当前 SQLite 表包括 `reports`、`resumes`、`applications`、`application_events`、`interview_sessions`、`interview_answers`、`resume_optimization_runs`、`optimization_feedback`、`stories`、`settings` 等业务表。
- 当前没有 `users`、`auth_sessions`、`auth_events` 等账号会话表。
- 当前没有 `middleware.ts` 或其他全局路由保护。

## 推荐鉴权策略

第一版不要直接做复杂 SaaS 用户体系。建议采用：

```text
本地 owner 账号
  -> 密码使用 Node crypto scrypt 哈希
  -> 登录后生成随机 session token
  -> Cookie 只保存原始 session token
  -> SQLite 只保存 token_hash
  -> API 每次通过 Cookie 校验 session
  -> 页面通过 Cookie gate + /api/auth/me 校验
```

理由：

- 不需要短信供应商和 OAuth 应用配置，能快速落地真实鉴权。
- 不引入 `bcrypt` / `argon2` native 依赖，降低 Windows 本地构建风险。
- 不把 JWT secret、长期 token 或明文密码放进前端。
- 和当前 `better-sqlite3` + 本地 SQLite 架构一致。
- 能先解决“未登录不能访问本地数据”的核心需求。

## 22. Auth 数据层与密码工具

### 目标

建立真实鉴权所需的最小后端基础，但不改 UI、不保护业务 API。

### 建议新增数据表

```sql
CREATE TABLE IF NOT EXISTS users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  email TEXT UNIQUE,
  phone TEXT UNIQUE,
  display_name TEXT NOT NULL DEFAULT 'New Era User',
  password_hash TEXT NOT NULL,
  password_salt TEXT NOT NULL,
  password_params_json TEXT NOT NULL DEFAULT '{}',
  role TEXT NOT NULL DEFAULT 'owner',
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  last_login_at TEXT
);

CREATE TABLE IF NOT EXISTS auth_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  token_hash TEXT NOT NULL UNIQUE,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  last_seen_at TEXT NOT NULL DEFAULT (datetime('now')),
  revoked_at TEXT,
  user_agent TEXT,
  FOREIGN KEY(user_id) REFERENCES users(id)
);

CREATE TABLE IF NOT EXISTS auth_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER,
  event_type TEXT NOT NULL,
  detail_json TEXT NOT NULL DEFAULT '{}',
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY(user_id) REFERENCES users(id)
);
```

### 建议新增文件

- `src/lib/auth/password.ts`
- `src/lib/auth/session.ts`
- `src/lib/auth/cookies.ts`
- `src/lib/auth/current-user.ts`
- `src/lib/auth/types.ts`

### 实现范围

- 在 `src/lib/server-db.ts` 增加幂等迁移。
- 用 `PRAGMA table_info(...)` 或 `CREATE TABLE IF NOT EXISTS` 保持兼容。
- 用 Node `crypto.scrypt` 或 `crypto.scryptSync` 做密码哈希。
- 用 `crypto.randomBytes(32)` 生成 session token。
- 只在数据库保存 `sha256(token)`，不保存原始 token。
- 增加创建 owner 用户、读取 owner 用户、创建 session、撤销 session、校验 session 的 DB 方法。
- 增加 Zod schema，例如注册、登录、修改密码输入。

### 不做

- 不做 API route。
- 不改 `/auth` 页面。
- 不保护业务 API。
- 不添加 `user_id` 到全部业务表。

### 验收标准

- `npm run lint` 通过。
- `npm run build` 通过。
- 本地启动时自动创建 `users`、`auth_sessions`、`auth_events` 表。
- 密码不会明文存储。

## 23. 注册/登录/登出 API

### 目标

提供真实后端鉴权接口，让前端可以创建本地 owner 账号、登录、登出、读取当前登录用户。

### API 建议

| 端点 | 方法 | 功能 |
|---|---|---|
| `/api/auth/register` | POST | 创建第一个本地 owner 账号 |
| `/api/auth/login` | POST | 校验密码并创建 session |
| `/api/auth/logout` | POST | 撤销当前 session 并清除 Cookie |
| `/api/auth/me` | GET | 返回当前登录用户和 session 状态 |

### Cookie 建议

```text
name: new_era_session
httpOnly: true
sameSite: lax
secure: production only
path: /
maxAge: 7 days
```

### 实现范围

- 所有 route handler 使用 `runtime = "nodejs"` 和 `dynamic = "force-dynamic"`。
- 注册只允许在当前没有 owner 用户时执行。
- 登录成功后写入 `auth_sessions`，设置 HttpOnly Cookie。
- 登出时撤销当前 session，并清除 Cookie。
- `/api/auth/me` 只返回安全字段：`id`、`displayName`、`email`、`phone`、`role`、`createdAt`、`lastLoginAt`。
- 所有错误使用 `apiSuccess` / `apiError` 项目既有风格。
- 错误文案使用中文，并区分：
  - `OWNER_ALREADY_EXISTS`
  - `INVALID_CREDENTIALS`
  - `SESSION_EXPIRED`
  - `UNAUTHENTICATED`

### 不做

- 不保护其他业务 API。
- 不改页面。
- 不做验证码发送。
- 不做 OAuth。

### 验收标准

- 无账号时可以注册 owner。
- 已有 owner 后再次注册返回明确错误。
- 正确密码可登录并设置 Cookie。
- 错误密码不会创建 session。
- 登出后 `/api/auth/me` 返回未登录。
- `npm run lint` 和 `npm run build` 通过。
- 用脚本或 PowerShell smoke 覆盖注册、登录、me、登出。

```text
- route handler 使用 nodejs runtime、force-dynamic、Zod 校验、apiSuccess/apiError。

```

## 24. API 鉴权保护

### 目标

让本地业务数据接口默认要求登录，未登录请求不能读取或写入报告、简历、投递、画像、面试、故事库、导出等数据。

### 需要保护的 API 范围

- `/api/reports`
- `/api/reports/[id]`
- `/api/resumes`
- `/api/resumes/*`
- `/api/resume/*`
- `/api/applications`
- `/api/applications/*`
- `/api/career-profile`
- `/api/career-profile/suggestions`
- `/api/analytics`
- `/api/assistant`
- `/api/interview/*`
- `/api/stories`
- `/api/settings`
- `/api/export`
- `/api/data/clear`
- 其他会读取或写入本地个人数据的 API。

### 可以放行的 API

- `/api/auth/*`
- `/api/health`
- 静态资源和 Next.js 内部资源。

### 实现范围

- 新增统一 helper，例如 `requireAuth()` 或 `getCurrentUserOrThrow()`。
- 在业务 API route handler 顶部调用鉴权 helper。
- 未登录统一返回 401。
- 已过期或 revoked session 统一返回 401，并可清理 Cookie。
- 不改变业务 API 的成功响应结构。
- 不把用户敏感信息写入业务响应。

### 技术注意

Next.js middleware 不能直接依赖 `better-sqlite3`。因此 API 真实鉴权应发生在 Node route handler 中，而不是在 middleware 中读取 SQLite。

### 不做

- 不在这一步改页面跳转。
- 不在这一步改 `/auth` UI。
- 不给所有业务表加 `user_id`。
- 不做多用户数据隔离。

### 验收标准

- 未登录请求受保护 API 返回 401。
- 登录后请求受保护 API 正常。
- `/api/auth/*` 和 `/api/health` 不被误拦截。
- 现有 API 的业务逻辑和响应字段不回退。
- `npm run lint` 和 `npm run build` 通过。

## 28. 鉴权验收与回归清单

### 目标

补齐鉴权相关自动化和人工验收，避免登录保护破坏已有核心求职流程。

### 建议新增或更新

- `scripts/smoke-auth.mjs`
- `docs/auth-acceptance-checklist.md`
- 可选：把 auth smoke 纳入 `npm run smoke`。

### API smoke 场景

- 首次注册 owner。
- 重复注册返回错误。
- 正确密码登录。
- 错误密码登录失败。
- 登录后 `/api/auth/me` 返回用户。
- 登出后 `/api/auth/me` 返回 401。
- 未登录访问受保护业务 API 返回 401。
- 登录后访问受保护业务 API 正常。

### 浏览器验收场景

- 未登录访问 `/applications` 跳转 `/auth`。
- 注册后进入首页。
- 登出后回到 `/auth`。
- 登录后回到原 `next` 页面。
- 错误密码显示错误，不跳转。
- 移动端 `/auth` 表单不溢出。
- 核心页面 `/evaluate`、`/resume`、`/applications` 登录后仍可用。

### 文案验收

- 不再声称当前只是 Demo 登录。
- 不暗示已发送短信验证码。
- 不暗示已接入微信/GitHub OAuth。
- 不暗示有云端同步、多用户 SaaS、招聘平台授权。
- 本地数据边界说明仍清晰。

### 不做

- 不新增长篇产品说明页。
- 不重构全部测试体系。
- 不把 auth smoke 做成复杂 e2e 框架，除非已有工具链要求。

### 验收标准

- `npm run lint` 通过。
- `npm run build` 通过。
- auth smoke 通过。
- 关键浏览器路径通过。
- 现有核心功能没有因为 401 或跳转逻辑被误伤。
- 形成可重复执行的验收清单。

## v2.1 完成标准

- 当前 `/auth` 不再只是 Demo，而是真实注册/登录入口。
- 本地 SQLite 中有 owner 账号和 session 记录。
- 密码不明文存储。
- Cookie 使用 HttpOnly。
- 未登录不能访问本地个人数据 API。
- 未登录不能进入主要产品页面。
- 已登录用户可以正常使用 JD 评估、简历、投递、画像、分析、Assistant、面试等既有功能。
- 设置页能展示真实账号状态，并支持修改密码和退出登录。
- `npm run lint` 和 `npm run build` 通过。
- auth smoke 和浏览器验收通过。
- 没有引入短信、OAuth、云同步、多用户 SaaS、招聘平台授权等超范围承诺。

## 一个步骤一个新对话的执行方式

每个新对话只实现一个编号，不要在同一对话顺带实现后续步骤。

推荐依次创建 7 个新对话：

1. `New Era v2.1 - Step 22 - Auth 数据层与密码工具`
2. `New Era v2.1 - Step 23 - 注册登录登出 API`
3. `New Era v2.1 - Step 24 - API 鉴权保护`
4. `New Era v2.1 - Step 25 - 页面访问控制`
5. `New Era v2.1 - Step 26 - 登录注册页面真实接入`
6. `New Era v2.1 - Step 27 - 账号安全设置`
7. `New Era v2.1 - Step 28 - 鉴权验收与回归清单`

每个对话的基础要求：

- 先阅读 `evolution/v2.1-auth-start-stage.md` 和当前步骤相关文件。
- 只实现当前编号，后续步骤不要提前做。
- 复用现有 Next.js、React、SQLite、Zod、`apiSuccess` / `apiError`、AppShell、UI 组件和设计风格。
- 新增 SQLite 结构必须兼容已有数据，迁移必须幂等。
- 所有输入输出都要经过 Zod 或等价 schema 校验。
- 涉及用户长期数据或写操作时，必须可查看、可确认、可撤销或可编辑。
- 不接招聘平台授权，不自动投递，不自动联系 HR。
- 不做短信、OAuth、云同步、多用户 SaaS，除非用户在后续步骤中明确改变范围。
- 完成后运行 `npm run lint` 和 `npm run build`。
- 涉及页面时做浏览器验收，并在最终回复中说明验收结果。
