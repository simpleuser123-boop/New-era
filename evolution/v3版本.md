# New Era v3 Agent 架构升级开发顺序

> 目标：把 New Era 当前的单 Assistant MVP 升级为可控、可验收的多 Agent 求职助手系统。v3 重点补齐工具注册表、子 Agent 路由、服务端 ReAct Loop、流式可视化、确认式动作和可管理记忆。每个编号都应拆成一个独立新对话执行，不要在同一个对话里顺带实现后续步骤。

## 范围边界

- 继续保持 local-first：业务数据优先存储在本地 `data/new-era.db`。
- 不做招聘网站自动爬取。
- 不做真实招聘平台 API 授权。
- 不做自动投递，不模拟已向外部平台提交简历。
- 不做自动联系 HR、自动发邮件、自动站内信。
- 不做支付、订阅、多用户 SaaS。
- 不做公开分享链接。
- 不做无人监管的自治任务队列。
- 不做无限长期记忆，所有记忆和画像都必须可查看、可编辑、可删除。
- Agent 可以辅助决策，但所有写操作必须让用户确认。
- Action 工具第一版只生成待确认动作，不直接写库；确认入口必须走明确的 API 和 UI。
- Query 工具必须是只读操作，允许 Agent 反复调用也不能污染本地数据。

## 当前基础

v3 默认开始前已经具备 v2 和 v2.1 的主要能力：

- JD 评估、SSE 流式评估、风险扫描、报告持久化。
- 简历解析、简历保存、默认简历、方向标签、软归档。
- 简历和 JD 匹配，能输出匹配度、技能覆盖和缺口建议。
- Tracker 完整工作台：投递状态、优先级、渠道、时间线、逾期跟进。
- Career DNA 求职画像：目标方向、目标岗位、城市偏好、优势、短板、底线。
- Analytics 求职分析：漏斗、方向、风险、技能缺口、Tracker 复盘、Career DNA 匹配。
- Assistant Agent MVP：`/assistant` 和 `POST /api/assistant`，支持本地数据问答、引用来源和待确认动作。
- Interview Coach、Resume Judge、数据导出与隐私治理。
- 本地 owner 登录鉴权和 API 保护。

## v3 开发顺序总览

| 顺序 | 功能 | 类型 | 目标产出 |
|---:|---|---|---|
| 29 | Agent 工具契约与注册表 | 架构基座 | `ToolDefinition`、Query/Action 分类、白名单执行门禁 |
| 30 | 本地 Query Tools 迁移 | 只读工具 | 把 Assistant 现有本地读取能力迁移成可注册工具 |
| 31 | 子 Agent Registry 与意图路由 | 多 Agent 基础 | Tracker/Evaluate/Resume/Interview/Career/General 子 Agent |
| 32 | 确认式 Action Proposals | 写入安全 | Agent 只生成待确认动作，确认后才调用既有 API |
| 33 | 服务端 Agent ReAct Loop | 执行循环 | `/api/agent/run`、工具调用循环、质量门控、SSE 事件 |
| 34 | Assistant 流式 Agent UI | 前端升级 | 阶段状态、工具调用卡片、引用来源、可取消/重试 |
| 35 | Agent 会话与可控记忆 | 上下文治理 | 会话列表、摘要、记忆查看/删除/导出，不做无限记忆 |
| 36 | Agent 验收与回归清单 | 工程质量 | API smoke、浏览器验收、误导文案和安全边界检查 |

## 依赖关系

```text
29 Agent 工具契约与注册表
  -> 30 本地 Query Tools 迁移
      -> 31 子 Agent Registry 与意图路由
          -> 32 确认式 Action Proposals
              -> 33 服务端 Agent ReAct Loop
                  -> 34 Assistant 流式 Agent UI
                      -> 35 Agent 会话与可控记忆
                          -> 36 Agent 验收与回归清单
```

- 29 先建立工具和执行边界，否则后续多 Agent 容易变成散落的提示词和函数调用。
- 30 只迁移只读工具，先保证 Agent 可以安全读取本地上下文。
- 31 在工具稳定后做子 Agent 路由，避免每个 Agent 私自拼上下文。
- 32 在 ReAct Loop 前先明确写入动作的确认机制，避免工具循环里误写数据。
- 33 负责真正的服务端 Agent Loop，前端只消费 SSE 事件。
- 34 把用户可见体验从一次性回答升级为阶段化、可解释的 Agent 工作流。
- 35 只做可管理记忆，不做黑盒长期记忆。
- 36 最后固化验收脚本和安全边界。

## 29. Agent 工具契约与注册表

### 目标

新增 Agent 工具系统的底层契约，建立统一的工具定义、注册、白名单和执行结果格式。第一步不接 LLM，不改 `/assistant` 行为，只把工具底座搭好。

### 核心设计

建议新增目录：

```text
src/lib/agent/
  tools/
    types.ts
    registry.ts
    index.ts
  types.ts
```

核心类型建议：

```ts
type ToolCategory = "query" | "action";

type ToolDefinition<TParams = unknown> = {
  name: string;
  displayName: string;
  description: string;
  category: ToolCategory;
  parameters: Record<string, ToolParameter>;
  handler: (params: TParams, context: AgentToolContext) => Promise<ToolResult>;
};

type ToolResult = {
  ok: boolean;
  data?: unknown;
  llmSummary: string;
  citations?: AssistantCitation[];
  action?: AssistantAction;
  error?: {
    code: string;
    message: string;
    recoverable: boolean;
  };
};
```

### 必须做

1. 新增 Agent 工具相关类型。
2. 新增 `ToolRegistry`，支持 `register()`、`get()`、`list()`、`execute()`。
3. 支持 `setActiveToolNames()` / `clearActiveToolNames()` 白名单门禁。
4. `execute()` 必须在工具调用前检查白名单。
5. Query / Action 分类必须进入类型和执行结果。
6. 所有工具参数必须通过 Zod 或等价 schema 校验。
7. 新增最小单元测试或脚本 smoke，覆盖注册、执行、白名单拒绝、参数错误。

### 不做

- 不接 DeepSeek。
- 不做 ReAct Loop。
- 不改 `/assistant` 页面。
- 不做子 Agent。
- 不做任何写库 Action 工具。
- 不做 Step 30-36。

### 验收标准

- 工具注册表可以注册和执行一个测试 Query 工具。
- 未授权工具在白名单开启时会被拒绝。
- 参数错误返回受控错误，不抛出未处理异常。
- `npm run lint` 和 `npm run build` 通过。

## 30. 本地 Query Tools 迁移

### 目标

把当前 Assistant MVP 里散落在 `assistant-tools.ts` 的本地读取能力迁移为可注册 Query Tools。此步骤只做只读工具，不做 Action，不改变现有 `/assistant` 返回逻辑。

### Query 工具范围

第一版建议包含：

- `search_reports`
- `get_report_detail`
- `list_applications`
- `get_application_detail`
- `get_analytics_overview`
- `get_career_profile`
- `list_resumes`
- `get_default_resume`
- `build_interview_prep_context`

### 工具原则

- Query 工具只能读本地 SQLite / settings / 既有聚合方法。
- Query 工具不能写库、不能修改 localStorage、不能触发外部网络。
- 每个工具必须返回结构化 `data` 和给 LLM 使用的 `llmSummary`。
- 每个工具应尽量返回 `citations`，供前端和最终回答展示来源。

4. 新增 Query Tools 统一导出和注册入口。
5. 为每个工具定义参数 schema。
6. 增加 smoke，逐个执行主要 Query Tools。

### 不做

- 不做 Action 工具。
- 不做子 Agent。
- 不做 ReAct Loop。
- 不把 `/api/assistant` 切到工具系统。
- 不做 Step 31-36。

### 验收标准

- 空数据状态返回受控摘要，不抛错。
- 现有 `/api/assistant` 行为不回归。
- `npm run lint` 和 `npm run build` 通过。

## 31. 子 Agent Registry 与意图路由

### 目标

新增多 Agent 的注册和路由层。此步骤只决定“该由哪个子 Agent 回答、它能用哪些工具、它的系统提示词是什么”，不做 ReAct Loop，不流式调用工具。

### 子 Agent 建议

| Agent ID | 名称 | 主要职责 | 工具范围 |
|---|---|---|---|
| `tracker-agent` | Tracker Agent | 投递状态、跟进优先级、逾期提醒、渠道复盘 | applications / analytics |
| `evaluate-agent` | Evaluate Agent | 报告解释、岗位风险、评分依据、JD 判断 | reports / analytics / career |
| `resume-agent` | Resume Agent | 默认简历、技能缺口、Resume Judge 建议解释 | resumes / reports / analytics |
| `interview-agent` | Interview Agent | 面试准备、问题生成上下文、STAR 故事建议 | interview prep / reports / resumes |
| `career-agent` | Career Agent | Career DNA、方向选择、目标岗位匹配 | career / analytics / applications |
| `general-agent` | General Agent | 兜底问答和跨域总结 | 只读综合工具 |

### 路由原则

- 使用确定性规则优先：关键词、上下文 ID、页面来源。
- 允许后续再加入 LLM intent classifier，但第一版不要依赖它。
- 每个 Agent 必须声明 `toolNames` 白名单。
- 每个 Agent 的 prompt 必须包含边界：不自动投递、不外部联系、不直接写入。

## 32. 确认式 Action Proposals

### 目标

把 Agent 写入意图统一变成“待确认动作”。此步骤可以新增 Action Proposal 类型、存储和确认 API，但 Agent 本身仍不能直接写入业务数据。

### Action 范围

第一版只允许：

- `propose_application_status_update`
- `propose_follow_up_event`
- `propose_career_profile_patch`
- `propose_resume_judge_feedback`
- `propose_interview_session_start`

### 安全原则

- Action 工具返回的是 proposal，不是执行结果。
- proposal 必须展示 payload、来源、生成原因、风险提示。
- 用户确认前不能修改 `applications`、`application_events`、`settings`、`resumes` 等业务表。
- 确认 API 必须重新校验 payload，不能信任 LLM 原始输出。
- 高风险或模糊动作必须要求用户去对应页面编辑，不直接确认执行。

### API 建议

```text
POST /api/agent/actions
GET /api/agent/actions?status=pending
POST /api/agent/actions/[id]/confirm
POST /api/agent/actions/[id]/dismiss
```

### 必须做

1. 新增 `AgentActionProposal` 类型和 schema。
2. 新增 `agent_action_proposals` SQLite 表，迁移必须幂等。
3. 新增 Action Proposal 工具定义。
4. 新增 proposal 创建、读取、确认、忽略 API。
5. 确认 API 只调用已有受控业务方法或 API 等价逻辑。
6. 增加 smoke，覆盖创建 proposal、确认、忽略、非法 payload 拒绝。

### 不做

- 不做自动投递。
- 不做外部联系。
- 不让 LLM 直接写业务表。
- 不做 ReAct Loop。
- 不做 Step 33-36。

### 验收标准

- Action Proposal 可创建、可查看、可确认、可忽略。
- 未确认 proposal 不改变业务表。
- 非法 payload 不会写库。
- `npm run lint` 和 `npm run build` 通过。

## 33. 服务端 Agent ReAct Loop

### 目标

新增服务端 Agent 执行循环。前端提交用户消息后，服务端选择子 Agent、构造上下文、调用模型、执行工具、检查结果质量，并通过 SSE 把阶段事件发给前端。

### API 建议

```text
POST /api/agent/run
```

请求：

```ts
type AgentRunRequest = {
  message: string;
  context?: {
    reportId?: number;
    applicationId?: number;
    resumeId?: number;
    sessionId?: number;
  };
};
```

SSE 事件建议：

```ts
type AgentSseEvent =
  | { type: "phase"; phase: "routing" | "understanding" | "executing" | "verifying" | "responding" }
  | { type: "agent_selected"; agentId: string; label: string }
  | { type: "tool_call"; name: string; params: Record<string, unknown> }
  | { type: "tool_result"; name: string; ok: boolean; summary: string; citations?: AssistantCitation[] }
  | { type: "action_proposed"; actionId: number; label: string }
  | { type: "text"; content: string }
  | { type: "done"; citations: AssistantCitation[]; warnings: string[] }
  | { type: "error"; error: { code: string; message: string } };
```

### Loop 原则

- API Key 和完整 prompt 只在服务端。
- 最大迭代数默认 5。
- 工具结果进入下一轮上下文前必须截断。
- 连续工具失败达到阈值后停止，并给出可理解错误。
- 空结果最多自动重试 1-2 次，不无限循环。
- Action 工具只能创建 proposal。
- 如果模型不稳定支持原生 function calling，可以先用严格 JSON tool call 协议，但必须在服务端校验工具名和参数。

### 必须做

1. 新增 `src/lib/agent/loop/`。
2. 新增 `/api/agent/run` SSE route。
3. 集成子 Agent 路由、工具注册表和白名单。

## 35. Agent 会话与可控记忆

### 目标

新增 Agent 会话和可控记忆。记忆用于改善跨轮对话体验，但必须可查看、可编辑、可删除、可导出，不做黑盒无限长期记忆。

### 存储建议

```text
agent_sessions
  id
  title
  messages_json
  memory_digest
  created_at
  updated_at

agent_memory_items
  id
  session_id
  memory_type      -- working_digest | episodic_summary | user_preference
  content
  source
  created_at
  updated_at
  deleted_at
```

### 记忆原则

- Working memory：当前会话最近 N 轮，由前端或服务端传入，不持久化为长期事实。
- Memory digest：规则或 LLM 摘要，但必须展示给用户。
- User preference：只有用户确认后才能进入长期偏好。
- 记忆删除后不得继续注入 prompt。
- 导出数据必须包含会话、记忆和 action proposal。

### API 建议

```text
GET /api/agent/sessions
POST /api/agent/sessions
GET /api/agent/sessions/[id]
PATCH /api/agent/sessions/[id]
DELETE /api/agent/sessions/[id]
GET /api/agent/memory
PATCH /api/agent/memory/[id]
DELETE /api/agent/memory/[id]
```
