# New Era v1.2 开始阶段开发顺序

> 目标：在 v1.1 已具备本地 `applications` 表和“加入投递清单”动作的基础上，补齐不依赖招聘平台授权的本地投递管理闭环。v1.2 开始阶段只覆盖 9-13，不涉及招聘网站自动爬取、真实岗位授权、自动投递、公网部署、支付订阅。

## 范围边界

- 不做招聘网站自动爬取。
- 不做真实岗位平台 API 授权。
- 不做自动投递，不模拟已向外部平台提交简历。
- 投递状态全部由用户在 New Era 内手动维护。
- 不做公网分享链接。
- 不做支付、订阅、多用户 SaaS。
- 数据继续优先存储在本地 SQLite。

## 当前基础

v1.1 已经为本阶段提供以下基础：

- 本地 SQLite 已有 `applications` 表。
- 已有 `GET /api/applications` 和 `POST /api/applications`。
- 已有 `GET /api/applications/[id]`。
- 已有最小 `PATCH /api/applications/[id]`，可更新 `to_apply / applied / abandoned`。
- 报告页可将报告加入本地投递清单。
- `applications` 已包含 `notes`、`tags_json`、`source_url` 字段。

## 开发顺序总览

| 顺序 | 功能 | 类型 | 目标产出 |
|---:|---|---|---|
| 9 | 新增投递管理页面入口 | 页面基础 | `/applications` 页面、侧边栏入口、空状态 |
| 10 | 投递清单列表与筛选 | 主流程 | 本地投递记录列表、搜索、状态筛选 |
| 11 | 扩展投递状态并支持手动流转 | 主流程 | 面试、Offer、未通过等本地状态管理 |
| 12 | 投递详情与跟进信息 | 主流程 | 备注、标签、原岗位链接、下次跟进时间 |
| 13 | 首页投递漏斗与本地统计 | 数据反馈 | 首页展示真实本地投递统计 |

## 依赖关系

```text
9 页面入口
  -> 10 列表与筛选
    -> 11 状态流转
      -> 12 详情与跟进
        -> 13 首页统计
```

- 9-13 建议按顺序执行，不要并行修改。
- 11 会修改状态枚举、schema、API 和页面状态展示。
- 12 会继续修改 `applications` 表、schema、API 和页面组件。
- 13 应在状态模型稳定后再接首页统计，避免重复调整统计口径。

## 9. 新增投递管理页面入口

### 目标

新增一个用户可进入的本地投递管理页面，把 v1.1 已有但不可见的 `applications` 数据能力转成正式产品入口。

### 实现范围

- 新增路由 `/applications`。
- 新增投递管理页面组件，例如：
  - `src/app/applications/page.tsx`
  - `src/components/features/applications/ApplicationManagementPage.tsx`
  - `src/components/features/applications/ApplicationWorkspace.tsx`
- 在侧边栏增加“投递管理”入口。
- 页面接入现有 `AppShell`，导航高亮正确。
- 先完成页面骨架：
  - 页面标题
  - 简短说明
  - 投递清单内容区
  - 加载状态
  - 空状态
  - 加载失败状态与重试
- 空状态提供“去评估岗位”按钮，跳转 `/evaluate`。

### 不做

- 不做外部招聘网站跳转后的自动提交。
- 不做列表筛选和状态流转，留到步骤 10、11。
- 不做复杂看板拖拽。

### 验收标准

- 侧边栏可以进入 `/applications`。
- `/applications` 页面刷新后仍能正常打开。
- 没有投递记录时展示清晰空状态。
- API 失败时展示错误状态，并允许重试。
- 页面在桌面端和窄屏下无明显溢出。

### 生成系统提示词要点

- 目标文件：`AppSidebar.tsx`、新增 `/applications` 页面和 applications 组件目录。
- 要求：先完成页面入口和状态骨架，读取现有 `GET /api/applications`，不扩展数据库。
- 验收：导航可进入，加载、空数据、错误三类状态可用。

## 10. 投递清单列表与筛选

### 目标

将本地投递记录展示为可扫描、可筛选的清单，让用户能快速找到需要处理的岗位。

### 实现范围

- 在 `/applications` 页面展示投递记录列表。
- 每条记录至少展示：
  - 公司
  - 岗位
  - 当前状态
  - 综合评分
  - 风险等级
  - 推荐简历名称
  - 最近更新时间
- 增加关键词搜索：
  - 匹配公司名称
  - 匹配岗位名称
- 增加状态筛选：
  - 全部
  - 待投递
  - 已投递
  - 已放弃
- 搜索与筛选可以叠加生效。
- 无匹配结果时展示重置筛选入口。
- 列表默认按 `updated_at DESC` 展示。

### 可选实现

- 数据量较小时可先在前端筛选。
- 如果调整 API，可为 `GET /api/applications` 增加可选参数：

```text
GET /api/applications?q=agent&status=to_apply&limit=50
```

### 不做

- 不做自动抓取岗位状态。
- 不做分页器，开始阶段保留 `limit` 即可。
- 不做批量编辑。

### 验收标准

- 从报告页加入的记录会出现在 `/applications`。
- 搜索公司或岗位关键词后列表正确过滤。
- 状态筛选与搜索同时使用时结果正确。
- 清空筛选后恢复完整清单。
- 风险等级和评分为空时页面不会报错。

### 生成系统提示词要点

- 目标文件：applications 页面组件，必要时调整 `GET /api/applications`。
- 要求：优先复用现有 API 和类型，不改状态枚举。
- 验收：列表、搜索、筛选、无结果状态可用。

## 11. 扩展投递状态并支持手动流转

### 目标

把最小投递记录升级为本地 Tracker。用户可以手动维护求职进度，但系统不连接招聘平台，也不声称已经替用户投递。

### 建议状态枚举

| 状态 | 中文展示 | 说明 |
|---|---|---|
| `to_apply` | 待投递 | 已加入清单，尚未手动投递 |
| `applied` | 已投递 | 用户已在外部渠道完成投递 |
| `interviewing` | 面试中 | 已进入面试流程 |
| `offer_received` | 已获 Offer | 已收到 Offer |
| `rejected` | 未通过 | 已明确未通过 |
| `abandoned` | 已放弃 | 用户主动停止推进 |

### 实现范围

- 扩展 `ApplicationStatus` 类型和 Zod schema。
- 更新 `PATCH /api/applications/[id]` 校验与错误文案。
- 在 `/applications` 列表中提供手动状态更新操作。
- 状态更新成功后立即刷新对应记录。
- 状态更新失败时保留原状态并显示反馈。
- 将“已投递”明确为用户手动确认，不绑定外部平台动作。
- 不要求数据库迁移：当前 `status` 字段为 `TEXT`，可直接兼容扩展枚举。

### 交互建议

- 开始阶段使用状态下拉菜单或操作菜单。
- 可以用分段筛选查看不同状态。
- 暂不引入拖拽库，减少页面复杂度。

### 不做

- 不做平台状态同步。
- 不做自动流转。
- 不做邮件或日历读取。
- 不做投递状态历史日志。

### 验收标准

- 用户可将记录从“待投递”改为“已投递”。
- 用户可继续改为“面试中”“已获 Offer”“未通过”或“已放弃”。
- 刷新页面后状态保持不变。
- 非法状态会被 API 拒绝。
- 页面没有任何“已替你投递成功”的误导文案。

### 生成系统提示词要点

- 目标文件：`types.ts`、`schemas.ts`、`server-db.ts`、applications API、applications 页面组件。
- 要求：只做本地手动状态流转，不接外部授权，不引入拖拽依赖。
- 验收：六类状态可保存、可刷新恢复、非法输入被拒绝。

## 12. 投递详情与跟进信息

### 目标

让用户为每条投递记录补充自己的判断和下一步动作，使投递清单不仅能看状态，也能承接实际跟进。

### 建议新增字段

```sql
ALTER TABLE applications ADD COLUMN next_follow_up_at TEXT;
```

已有字段继续使用：

- `notes`
- `tags_json`
- `source_url`

### 实现范围

- 自动迁移兼容已有 SQLite 数据。
- 更新 `ApplicationRow`、`ApplicationDto`、schema 和 DB 映射。
- 扩展 `PATCH /api/applications/[id]`，支持部分更新：
  - `status`
  - `notes`
  - `tags`
  - `sourceUrl`
  - `nextFollowUpAt`
- 在 `/applications` 中增加详情抽屉或详情区。
- 支持编辑：
  - 备注
  - 本地标签
  - 原岗位链接
  - 下次跟进时间
- 原岗位链接仅由用户手动填写或从报告记录带入。
- 链接打开前由用户主动点击，不做自动跳转。

### 不做

- 不做日历授权。
- 不做系统级提醒推送。
- 不做自动发送消息。
- 不做招聘平台联系人同步。
- 不做富文本编辑器。

### 验收标准

- 老数据库自动补齐 `next_follow_up_at`，原有记录不丢失。
- 用户可以保存备注、标签、原岗位链接和下次跟进时间。
- 刷新页面后编辑内容仍然存在。
- 留空字段可以正常保存。
- 非法 URL 或过长文本会被 schema 拒绝并显示友好提示。

### 生成系统提示词要点

- 目标文件：`server-db.ts`、`schemas.ts`、`types.ts`、applications API、applications 页面组件。
- 要求：兼容老 SQLite；PATCH 使用部分更新；不做外部服务授权。
- 验收：详情信息可编辑、可保存、刷新后可恢复。

## 13. 首页投递漏斗与本地统计

### 目标

将首页已有的求职漏斗从 Demo 估算升级为本地真实统计，让用户能看到自己已经评估、加入清单和推进中的岗位数量。

### 建议统计口径

| 指标 | 数据来源 |
|---|---|
| 已评估 | `reports` 总数 |
| 高匹配 | `reports` 中综合分达到阈值的数量 |
| 待投递 | `applications.status = 'to_apply'` |
| 已投递 | `applications.status = 'applied'` |
| 面试中 | `applications.status = 'interviewing'` |
| 已获 Offer | `applications.status = 'offer_received'` |
| 未通过 | `applications.status = 'rejected'` |
| 已放弃 | `applications.status = 'abandoned'` |

### 实现范围

- 增加本地统计 DB 方法，例如 `getApplicationStats`。
- 增加统计 API，例如：

```text
GET /api/applications/stats
```

- 首页读取真实统计数据。
- 更新首页漏斗或统计卡：
  - 已评估
  - 高匹配
  - 待投递
  - 已投递
  - 面试中
  - 已获 Offer
- 在数据为空时展示零值或明确空状态。
- 统计接口失败时允许降级展示，不影响首页其他区域。
- 首页提供进入 `/applications` 的入口。

### 不做

- 不做复杂 BI 仪表盘。
- 不做按月、按渠道、按简历方向的高级分析。
- 不做平台转化率对账。
- 不做自动生成求职策略。

### 验收标准

- 修改投递状态后，刷新首页统计会同步变化。
- 新加入投递清单后，“待投递”数量增加。
- 改为“面试中”后，对应状态数量变化正确。
- 数据为空和接口失败时首页仍可用。
- 所有统计都来自本地数据库，不依赖外部授权。

### 生成系统提示词要点

- 目标文件：`server-db.ts`、applications stats API、`HomeDashboard.tsx`。
- 要求：使用本地聚合查询；失败时局部降级；不做高级分析。
- 验收：投递状态改变后首页统计口径正确。

## v1.2 开始阶段完成标准

- 侧边栏有正式“投递管理”入口。
- `/applications` 能展示本地投递清单。
- 用户能搜索、筛选投递记录。
- 用户能手动维护待投递、已投递、面试中、已获 Offer、未通过、已放弃状态。
- 用户能保存备注、标签、原岗位链接和下次跟进时间。
- 首页展示来自本地 SQLite 的投递漏斗统计。
- 不引入招聘平台授权依赖。
- 不出现“系统已经自动替用户投递”的误导文案。
- 不破坏现有评估、报告、风险扫描、简历解析、简历匹配、历史报告流程。

## 一个步骤一个新对话的执行方式

每个新对话只实现一个编号，不要在同一对话顺带实现后续步骤。

推荐依次创建 5 个新对话：

1. `New Era v1.2 - Step 09 - 投递管理页面入口`
2. `New Era v1.2 - Step 10 - 投递清单列表与筛选`
3. `New Era v1.2 - Step 11 - 投递状态手动流转`
4. `New Era v1.2 - Step 12 - 投递详情与跟进信息`
5. `New Era v1.2 - Step 13 - 首页投递漏斗统计`

每个对话的基础要求：

- 先阅读本文件和 `evolution/v1.1-start-stage.md`。
- 只实现当前编号，后续步骤不要提前做。
- 先检查现有代码，复用已有类型、API、组件和设计系统。
- 兼容已有本地 SQLite 数据。
- 完成后运行 lint；涉及页面时做一次浏览器验收。
- 在最终回复中列出修改文件、验收结果和下一步应该开启的对话标题。
