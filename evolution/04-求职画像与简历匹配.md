# 04 — 求职画像与简历匹配

> 所属阶段：当前实现基线 · Career DNA + 简历匹配 + Resume Judge

---

## 1. 设计理念

New Era 的画像系统核心思路：**用 AI 技术栈图谱做匹配中枢，用 Career DNA 做个人求职边界。** 简历画像回答“我有什么能力”，Career DNA 回答“我想去哪、哪些条件不能接受”，Resume Judge 回答“这份简历面对这个目标岗该怎么改”。

```
简历文本
    │
    ▼
┌─────────────────────────────┐
│ AI 技术栈解析                │
│                             │
│ 提取:                        │
│  框架 (PyTorch, LangChain)  │
│  工具 (Milvus, HuggingFace) │
│  领域 (RAG, Agent, CV...)   │
│  经验层次 (入门/熟练/专家)    │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ AI 生态位定位                │
│                             │
│ LLM应用  ████████░░ 80%    │
│ AI Infra ██░░░░░░░░ 20%    │
│ 算法研究 ████░░░░░░ 40%    │
│ 具身智能 ░░░░░░░░░░ 0%     │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ JD 匹配                      │
│                             │
│ 简历生态位 vs JD 生态位       │
│ → 匹配度高 (同方向)           │

│ → 匹配度低 (不同方向)         │
└─────────────────────────────┘
              │
              ▼
┌─────────────────────────────┐
│ Career DNA / Resume Judge    │
│                             │
│ 目标方向、城市、薪资、边界     │

│ → 简历优化 / ATS 检查         │
└─────────────────────────────┘
```

当前实现补充（2026-06-11）：
- 简历数据保存在 `resumes`，支持方向、默认简历、软归档。
- Career DNA 保存在 `settings.career_profile`，建议必须用户确认并保存。
- Resume Judge 和 ATS 检查结果保存在 `resume_optimization_runs`。
- Analytics 和 Agent 会读取 Career DNA，但不能绕过用户确认写入长期画像。

## 设计思想

AI 从业者的简历有一个独特的特点：**技能高度可迁移但方向差异巨大**。一个精通 PyTorch 的工程师可以做 LLM 微调（算法研究方向），也可以做模型 serving 优化（AI Infra 方向），但因为缺乏 Agent 开发经验而在 LLM 应用层竞争力偏弱。传统简历匹配只看关键词交集，看不到这种"方向性匹配"——New Era 用 AI 生态位图谱解决了这个问题。

核心洞察：**匹配不是技能的集合运算，而是生态位的距离计算。** 同一份简历在 LLM 应用层是 80% 匹配，在 AI Infra 层可能是 40%——不是技能少了，是方向偏了。用户需要的不是"你的技能覆盖了 JD 的 65%"，而是"你的技能在这个方向上有多对口"。

## 2. AI 技术栈图谱

### 2.1 五层分类体系

```
AI 技术栈五层分类
═══════════════════════════════════════

Layer 1: 基础框架
  PyTorch, TensorFlow, JAX, NumPy, Pandas

Layer 2: 模型层
  Transformers, Diffusers, vLLM, TGI, ONNX Runtime

Layer 3: 应用框架
  LangChain, LlamaIndex, Semantic Kernel, CrewAI, AutoGen

Layer 4: 基础设施
  Milvus, Pinecone, Chroma, Weaviate, Qdrant (向量数据库)

  Ray, Dask (分布式计算)

Layer 5: 评估与监控
  LangSmith, LangFuse, Phoenix, RAGAS, DeepEval
```

### 2.2 五个 AI 细分方向

| 方向 | 核心技术栈 | 典型岗位 |
|------|-----------|---------|
| LLM 应用层 | LangChain, RAG, Agent, Prompt Engineering | AI 应用开发 / AI 产品经理 |
| AI Infra | 模型 serving, 向量数据库, GPU 集群, MLOps | AI 基础设施工程师 |
| 算法研究 | PyTorch, 模型训练, 论文, LoRA/RLHF | 算法研究员 / 大模型训练 |
| 具身智能 | ROS, 仿真环境, 机器人控制, 感知算法 | 机器人算法工程师 |
| AI 安全 | 对抗样本, 红队测试, 模型对齐, 可解释性 | AI 安全研究员 |

## 4. JD 匹配算法

### 4.1 匹配流程

```
简历生态位向量 [0.8, 0.2, 0.4, 0.0, 0.1]
       ×
JD 生态位向量   [0.9, 0.1, 0.2, 0.0, 0.0]
       =
方向匹配度: 0.92 (高度匹配，同为 LLM 应用层)
技能覆盖率: 75% (简历技能覆盖了 JD 要求的 75%)
综合匹配度: 方向匹配度 × 0.6 + 技能覆盖率 × 0.4 = 0.85
```

### 4.2 匹配度分级

| 匹配度 | 含义 | 建议 |
|--------|------|------|
| 0.8+ | 高度匹配 | 优先投递 |
| 0.6-0.8 | 中度匹配 | 可以投，注意补短板 |
| 0.4-0.6 | 低度匹配 | 跨方向投递，需突出可迁移技能 |
| <0.4 | 不匹配 | 不建议投递 |

### 4.3 缺口分析输出

```markdown
## 技能缺口分析

### 已匹配 (3/6)
- ✅ LangChain/LlamaIndex (项目 #2)
- ✅ RAG 系统设计 (项目 #1)

### 可快速弥补 (2/6)
```

## 5. 简历优化建议

基于 JD 要求 + 缺口分析，生成 3-5 条具体建议：

1. 项目描述强化：把"搭建了 RAG 系统"改为"设计并实现了企业级 RAG 系统，采用 LangChain + Milvus，日均处理 5000+ 查询，检索准确率 92%"
2. 技能关键词对齐：JD 要求"Agent 开发"，简历中有 LangChain 经验但未提及 Agent — 补充相关项目或学习经历
3. AI 生态位标注：简历开头增加一行"AI 方向：LLM 应用层，5 年经验，专注 RAG 和 Agent 开发"

## 6. 当前实现边界

| 能力 | 当前做法 | 边界 |
|------|----------|------|
| 简历画像 | SQLite `resumes` 保存结构化内容、技能、生态位 | 已归档简历不参与匹配和 ATS |
| Career DNA | SQLite `settings.career_profile` 保存目标、偏好、优势、短板 | 建议不自动保存，必须用户确认 |
| 方向标签 | 固定 AI 方向枚举 | 不做自由文本方向导致的比较混乱 |
| 默认简历 | 单一默认简历 | 不能有多个默认简历 |
| Resume Judge | 目标报告 / 手动 JD + 已保存简历 | 不直接改写简历正文，只给建议和反馈 |
| Analytics 复用 | 读取简历、Tracker、Career DNA 聚合分析 | 样本少时必须显示 caveat |
